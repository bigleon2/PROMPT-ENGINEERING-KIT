   115→   115→## MODE 3 : HYBRIDE — sélection automatique
   116→   116→**Usage** : mode par défaut, recommandé pour tous les projets.
   117→   117→**Principe** : analyse automatique de la complexité pour basculer entre CoT et Chaining.
   118→   118→
   119→   119→Algorithme de sélection :
   120→   120→1. Évaluer la complexité du projet (volume, nombre de composants, ambiguïté)
   121→   121→2. **Complexité « simple »** (projet mono-fichier, objectif clair, < 5 composants)
   122→   122→   → Mode Chaining (4 étapes)
   123→   123→3. **Complexité « moyen »** (projet multi-fichiers, quelques ambiguïtés, 5-10 composants)
   124→   124→   → Mode Chaining avec étape de clarification supplémentaire
   125→   125→4. **Complexité « complexe »** (projet multi-dossiers, contraintes fortes, > 10 composants)
   126→   126→   → Mode CoT complet (7 étapes)
   127→   127→5. L'utilisateur peut toujours forcer un mode spécifique via l'interface
   128→   128→
   129→   129→Note : en mode Hybride, la détection de complexité s'appuie sur l'arborescence uploadée,
   130→   130→le nombre de fichiers, la diversité des extensions, et la présence de patterns de projet
   131→   131→reconnus (package.json, pyproject.toml, docker-compose.yml, etc.).

   362→   362→- **Autorisé** : adapter au projet, inclure cas limites, varier les domaines
   363→   363→- **Interdit** : exemples génériques, mono-domaine, doublons entre fichiers
   364→   364→
   365→   365→## RÈGLE 7 : AUTOMATISATION ET VERSIONNAGE (nouveau v4.0)
   366→   366→**Description** : le pipeline automatisé doit être utilisé quand disponible.
   367→   367→- **Autorisé** : utiliser build_kit.py, validate_kit.py, tracer dans VERSION.yaml
   368→   368→- **Interdit** : ignorer les outils automatisés, générer manuellement si le script existe
   369→   369→
   370→   370→## Règle 8 : Détection de Contenu Trompeur (nouveau v4.1)
   371→   371→**Description** : tout contenu généré doit être vérifiable et ne pas induire en erreur.
   372→   372→- **Autorisé** : vérifier les allégations, signaler les risques, proposer des formulations alternatives honnêtes
   373→   373→- **Interdit** : générer du greenwashing, des fausses promesses, des statistiques inventées
   374→   374→
   375→   375→## Règle 9 : Boucle de Feedback Itérative (nouveau v4.1)
   376→   376→**Description** : les résultats de tests A/B doivent être intégrés comme signal d'amélioration.
   377→   377→- **Autorisé** : intégrer les métriques A/B dans les itérations, ajuster les prompts en conséquence
   378→   378→- **Interdit** : ignorer les données de feedback quantitatives disponibles
   379→   379→
   380→   380→## Gestion des Cas Limites
   381→   381→- **Projet incomplet** → kit avec placeholders `[À COMPLÉTER]` + demande de compléments
   382→   382→- **Contradictions** → identifier et proposer une résolution (cf. few_shot Exemple 2)
   383→   383→- **Informations ambiguës** → présenter plusieurs interprétations et demander validation
   384→   384→- **Input corrompu** → signaler l'erreur et proposer une alternative de format
   385→   385→- **Auto-référence** → le kit peut s'analyser lui-même ; éviter la boucle infinie
   386→   386→- **Génération circulaire** → détecter et interrompre si un kit généré génère un kit
   387→   387→- **Contenu trompeur** → activer la règle 8, proposer des reformulations vérifiables (cf. few_shot Exemple 6)
   388→   388→- **Feedback A/B non exploité** → activer la règle 9, intégrer les métriques dans la prochaine itération
