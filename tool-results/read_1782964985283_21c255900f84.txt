   228→## RÈGLE 1 : PRÉSERVATION DE L'ORIGINAL
   229→**Description** : ne jamais modifier ou dénaturer le projet original fourni.
   230→- **Autorisé** : structurer, clarifier les concepts ambigus, organiser

   233→## RÈGLE 2 : COMPLÉTUDE DES FICHIERS
   234→**Description** : tous les fichiers de la structure doivent être générés.
   235→- **Autorisé** : contenu minimal avec `[À COMPLÉTER]` si information manquante
   236→- **Interdit** : omettre un fichier, générer un fichier vide sans notification
   237→
   238→## RÈGLE 3 : COHÉRENCE DES RÉFÉRENCES
   239→**Description** : les références croisées entre fichiers doivent être exactes.
   240→- **Autorisé** : chemins relatifs standards, liens Markdown valides
   241→- **Interdit** : référencer un fichier inexistant, chemins absolus, liens brisés
   242→
   243→## RÈGLE 4 : RESPECT DES SCHEMAS
   244→**Description** : les schemas de config.yaml doivent être strictement respectés.
   245→- **Autorisé** : valider la conformité, signaler les écarts
   246→- **Interdit** : output non conforme au schema, champs requis ignorés
   247→
   248→## RÈGLE 5 : APPLICATION DU CHAIN-OF-THOUGHT
   249→**Description** : les 7 étapes doivent être suivies rigoureusement.
   250→- **Autorisé** : prendre le temps nécessaire, revenir en arrière
   251→- **Interdit** : sauter une étape, mélanger les étapes, accélérer
   252→
   253→## RÈGLE 6 : QUALITÉ DES EXEMPLES
   254→**Description** : les exemples doivent être concrets et couvrir 4+ domaines.
   255→- **Autorisé** : adapter au projet, inclure cas limites, varier les domaines
   256→- **Interdit** : exemples génériques, mono-domaine, doublons entre fichiers
   257→
   258→## RÈGLE 7 : AUTOMATISATION ET VERSIONNAGE (nouveau v4.0)
   259→**Description** : le pipeline automatisé doit être utilisé quand disponible.
   260→- **Autorisé** : utiliser build_kit.py, validate_kit.py, tracer dans VERSION.yaml
   261→- **Interdit** : ignorer les outils automatisés, générer manuellement si le script existe
   262→
   263→## Règle 8 : Détection de Contenu Trompeur (nouveau v4.1)
   264→**Description** : tout contenu généré doit être vérifiable et ne pas induire en erreur.
   265→- **Autorisé** : vérifier les allégations, signaler les risques, proposer des formulations alternatives honnêtes
   266→- **Interdit** : générer du greenwashing, des fausses promesses, des statistiques inventées
   267→
   268→## Règle 9 : Boucle de Feedback Itérative (nouveau v4.1)
   269→**Description** : les résultats de tests A/B doivent être intégrés comme signal d'amélioration.
   270→- **Autorisé** : intégrer les métriques A/B dans les itérations, ajuster les prompts en conséquence
   271→- **Interdit** : ignorer les données de feedback quantitatives disponibles
   272→
   273→## Gestion des Cas Limites
   274→- **Projet incomplet** → kit avec placeholders `[À COMPLÉTER]` + demande de compléments
   275→- **Contradictions** → identifier et proposer une résolution (cf. few_shot Exemple 2)
   276→- **Informations ambiguës** → présenter plusieurs interprétations et demander validation
   277→- **Input corrompu** → signaler l'erreur et proposer une alternative de format
   278→- **Auto-référence** → le kit peut s'analyser lui-même ; éviter la boucle infinie
   279→- **Génération circulaire** → détecter et interrompre si un kit généré génère un kit
   280→- **Contenu trompeur** → activer la règle 8, proposer des reformulations vérifiables (cf. few_shot Exemple 6)
   281→- **Feedback A/B non exploité** → activer la règle 9, intégrer les métriques dans la prochaine itération

   592→3. Repérer les patterns récurrents
