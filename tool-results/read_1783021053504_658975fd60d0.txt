    34→## 2. PHILOSOPHIE DE L'ÉCOSYSTÈME
    35→
    36→### Principe Fondamental
    37→**Performance-driven selection** : Le choix entre skill, agent spécialisé ou agent général est dicté par le gain de performance, pas par une hiérarchie rigide.
    38→
    39→### Modèle à Deux Couches
    40→```
    41→SKILL (protocole + connaissance domaine)
    42→    ↓
    43→AGENT SPÉCIALISÉ (exécution + outils)
    44→```
    45→
    46→- Le **skill** fournit le "comment" (protocole, templates, critères qualité)
    47→- L'**agent** fournit le "qui" (capacités spécialisées, outils, vitesse)
    48→
    49→### Principes Directeurs
    50→1. **Read before planning** — Toujours lire le projet avant de planifier
    51→2. **Performance-driven** — Sélectionner l'agent optimal pour chaque sous-tâche
    52→3. **Serial by default** — Exécution sérielle obligatoire sauf exception
    53→4. **Visible progress** — L'utilisateur sait toujours où on en est
    54→5. **Quality gates** — Validation après chaque phase
    55→
    56→### Exécution Sérielle par Défaut
    57→- **Toutes les tâches s'exécutent UNE À LA UNE** dans l'ordre défini
    58→- **Pas d'agents simultanés** sauf demande explicite + indépendance prouvée
    59→- **Quality gate entre phases** : vérifier les outputs avant de continuer
    60→- **Corrections en vol** : réévaluer si l'état du projet change significativement

   275→## 4. SKILL GEN-PLAN — Orchestration (v1.2.0)
   276→
   277→**Déclencheurs** : `gen-plan:`, `plan d'actions`, `orchestre`
   278→
   279→### 4.1 Modes
   280→
   281→| Mode | Déclencheur | Comportement |
   282→|------|-------------|--------------|
   283→| **TÂCHE** | `gen-plan: <description>` | Planifie et exécute une tâche spécifique |
   284→| **PROJET** | `gen-plan:correct-work(projet)` | Analyse complète, corrections, mise à jour prompt-master |
   285→| **CIBLE** | `gen-plan:correct-work(<cible>)` | Vérification/correction d'un élément spécifique |
   286→
   287→### 4.2 Protocole — 13 Étapes Séquentielles
   288→
   289→#### Étape 1 — Collecte et Analyse des Demandes
   290→**Objectif** : Relire toute la conversation pour extraire chaque demande explicite et implicite.
   291→- Relire tous les messages utilisateur (demandes initiales, specs, contraintes, corrections)
   292→- Identifier demandes explicites (ce que l'utilisateur a directement demandé)
   293→- Identifier demandes implicites (prérequis, effets de bord, conséquences logiques)
   294→- Lister chaque demande sans ambiguïté
   295→- **Méthode** : Lire conversation complète. Utiliser worklog.md si disponible.
   296→
   297→#### Étape 2 — Lecture et Analyse du Projet
   298→**Objectif** : Traverser tous les fichiers du projet pour comprendre structure, architecture, dépendances, état actuel.
   299→- **Principe** : Un plan sans connaissance du projet est générique et probablement inadéquat.
   300→- Structure fichiers : arborescence, configs (package.json, tsconfig, etc.)
   301→- Code source : chaque fichier significatif
   302→- Schéma base de données (si applicable)
   303→- Dépendances : imports inter-modules, dépendances externes
   304→- Configuration : fichiers config pertinents
   305→- Documentation : README, worklog, specs, fichiers référence
   306→- **Méthode** : Utiliser Read, Glob, Grep pour traverser chaque fichier significatif.
   307→
   308→#### Étape 3 — Identification de la Nature du Projet
   309→**Objectif** : Déterminer type projet, technologies, architecture, complexité.
   310→- Type projet : web app, API, script, document, analyse, skill, etc.
   311→- Technologies : framework, langage, base de données, build tools
   312→- Architecture : monolithe, micro-services, fullstack, frontend only, backend only
   313→- Complexité : nombre fichiers, profondeur arborescence, dépendances
   314→
   315→#### Étape 4 — Identification des Objectifs
   316→**Objectif** : Lister chaque objectif explicitement, sans ambiguïté ni omission.
   317→- Lister objectifs dérivés des demandes utilisateur
   318→- Lister objectifs implicites (prérequis, effets de bord, contraintes)
   319→- Vérifier que chaque objectif est mesurable (atteint ou non)
   320→- Supprimer doublons
   321→
   322→#### Étape 5 — Décomposition en Sous-tâches Adaptées
   323→**Objectif** : Décomposer chaque objectif en sous-tâches atomiques avec ordre logique d'exécution.
   324→- Pour chaque objectif, identifier sous-tâches atomiques
   325→- Ordonner logiquement (prérequis d'abord, puis exécution, puis validation)
   326→- Adapter au type projet :
   327→  - **Fullstack** : inclure vérification API frontend-backend, schéma DB, auth
   328→  - **Frontend only** : inclure vérification responsive, accessibilité, composants
   329→  - **Backend/API** : inclure vérification endpoints, validation, sécurité
   330→  - **Document/PDF** : inclure vérification contenu, layout, cohérence données
   331→  - **Script/automation** : inclure cas limites, robustesse, gestion erreurs
   332→
   333→#### Étape 6 — Détection des Dépendances
   334→**Objectif** : Identifier dépendances entre sous-tâches et contraintes de précédence.
   335→- Dépendances séquentielles (A doit finir avant B)
   336→- Dépendances parallèles (A et B peuvent tourner simultanément)
   337→- Dépendances conditionnelles (B tourne seulement si A réussit/échoue)
   338→- Vérifier absence de dépendances circulaires
   339→- **Méthode** : Construire graphe de dépendances. Identifier chemins critiques.
   340→
   341→#### Étape 7 — Priorisation
   342→**Objectif** : Classer sous-tâches par impact sur résultat final.
   343→
   344→| Priorité | Définition |
   345→|----------|------------||
   346→| **Critique** | Échec invalide le résultat entier |
   347→| **Importante** | Impact significatif sur qualité |
   348→| **Secondaire** | Améliore résultat mais non bloquant |
   349→
   350→#### Étape 8 — Estimation des Risques
   351→**Objectif** : Pour chaque sous-tâche, identifier risques potentiels et plans de fallback.
   352→- Complexité technique : connaissance spécialisée requise ?
   353→- Ambiguïté : clairement définie ou ouverte à interprétation ?
   354→- Dépendance externe : repose sur API, service, outil qui pourrait échouer ?
   355→- Fallback : que faire si sous-tâche échoue ? Contourner, reporter, simplifier ?
   356→
   357→#### Étape 9 — Structuration du Plan
   358→**Objectif** : Produire un plan d'actions formel, complet, adapté au projet.
   359→
   360→Le plan doit inclure :
   361→- **En-tête** : nature projet, objectifs, contraintes
   362→- **Liste ordonnée d'étapes** avec IDs (1, 2-a, 2-b, 3...)
   363→  - Pour chaque étape : objectif, fichiers concernés, dépendances, priorité
   364→  - Indication parallélisme où applicable
   365→  - Critères de validation par étape
   366→- **Carte dépendances** : résumé relations étapes
   367→- **Matrice risques** : risques et plans fallback
   368→
   369→#### Étape 10 — Validation du Plan
   370→**Objectif** : Checklist :
   371→- [ ] Le plan couvre-t-il toutes les demandes utilisateur ?
   372→- [ ] Y a-t-il des étapes manquantes ?
   373→- [ ] L'ordre d'exécution est-il logique (aucune dépendance violée) ?
   374→- [ ] Le plan est-il adapté au projet (pas générique) ?
   375→- [ ] Chaque étape a-t-elle un critère de validation clair ?
   376→- [ ] Les risques principaux sont-ils couverts par des plans fallback ?
   377→
   378→Si problème trouvé, retourner à l'étape pertinente et corriger.
   379→
   380→#### Étape 11 — Test et Correction Pré-Intégration
   381→**Objectif** : Tester chaque fichier généré ou modifié AVANT de l'intégrer à l'éco-système.
   382→
   383→Pour chaque fichier candidat à l'intégration :
   384→1. **Classifier le fichier** :
   385→   - **Skill** : Fichier avec YAML frontmatter (name, description) ou contenu définissant un protocole/routine réutilisable
   386→   - **Éco-système** : Fichier de configuration, prompt maître, contexte, mémoire, agent
   387→   - **Utilitaire** : Script, outil, fichier temporaire — ne pas intégrer
   388→2. **Tester le fichier** :
   389→   - **Skill (.md avec YAML)** : Vérifier YAML valide, contenu cohérent (> 200 chars), pas de placeholders
   390→   - **Python (.py)** : Compiler avec compile() ou ast.parse() pour vérifier syntaxe
   391→   - **Shell (.sh)** : Vérifier shebang, syntaxe avec bash -n, pas de commandes destructrices
   392→   - **Markdown (.md)** : Vérifier structure, absence de contenu tronqué
   393→   - **Configuration (.json, .yaml)** : Valider format, vérifier clés obligatoires
   394→3. **Corriger les anomalies détectées** immédiatement
   395→4. **Valider la cohérence entre fichiers** (références croisées valides)
   396→
   397→#### Étape 12 — Intégration Éco-Système (SPÉCIFIQUE z.ai)
   398→**Objectif** : Si les fichiers concernent des skills ou l'éco-système, les intégrer.
   399→
   400→**Prérequis** : Étape 11 (Test et Correction) doit être terminée avec succès.
   401→
   402→1. **Si le fichier est un Skill** :

   500→
   501→**Déclencheurs** : `vérifie ton travail`, `correct_work()`, `verify_work`
   502→
   503→**3 Modes** : PROJET | CIBLE | DIRECT
   504→

   506→Skill de vérification du travail réalisé. Alias : verify-work. Version v1.0.0.
   507→DÉCLENCHEURS :
   508→- "vérifie ton travail", "vérifie tes résultats", "vérifie ton code"
   509→- "correct_work", "verify_work"
   510→3 MODES D'APPEL :
   511→- PROJET : correct_work(projet) — vérification complète avec gestion du prompt-maître. Première analyse = intégralité du projet ; appels suivants = lecture du prompt-maître existant uniquement
   512→- CIBLE : correct_work(cible) — vérification ciblée sur un livrable spécifique, sans gestion du prompt-maître
   513→- DIRECT : correct_work() — vérification rapide sans analyse approfondie
   514→PROTOCOLE 5 ÉTAPES SÉQUENTIELLES :
   515→ÉTAPE 1 — Plan d'actions (appel à gen-plan) :
   516→Appelle le skill gen-plan pour générer un plan de vérification adapté au projet. Le plan sert de base pour les étapes 2 à 5. Le plan DOIT être spécifiquement adapté au type de projet vérifié (web, Python, PDF, script, etc.).
   517→ÉTAPE 2 — Erreurs et omissions :
   518→1. Relire les spécifications initiales et vérifier chaque exigence satisfaite
   519→2. Vérifier les données factuelles (noms, chemins, versions, tailles, counts)
   520→3. Vérifier la cohérence linguistique (pas de mélange de langues incohérent)
   521→4. Vérifier les fichiers de sortie (existent, lisibles, non vides)
   522→5. Vérifier les dépendances (imports, chemins, références croisées)
   523→6. Adapter la vérification au type de projet
   524→7. Corriger chaque erreur ou omission identifiée
   525→ÉTAPE 3 — Structure du code et conflits :
   526→1. Imports circulaires
   527→2. Conflits de noms (fonctions/classes/variables en double)
   528→3. Variables non initialisées ou utilisées avant définition
   529→4. Chemins en dur non portables
   530→5. Gestion des erreurs (cas d'erreur traités ou échec silencieux ?)
   531→6. Doublons de code à factoriser
   532→7. Convention de nommage cohérente
   533→8. Conflits logiques et conditions contradictoires — utiliser un TABLEAU DE KARNAUGH pour simplifier, détecter les contradictions, vérifier la couverture complète
   534→9. Corriger chaque problème identifié
   535→ÉTAPE 4 — Vérification des interactions :
   536→Adaptée au type de projet (fullstack, frontend, backend, document, script) :
   537→1. API frontend-backend : endpoints existent, params correspondent, réponses parsées, codes erreur gérés
   538→2. Props inter-composants : types corrects, callbacks bons args, pas de props manquantes
   539→3. State management : store expose données nécessaires, actions au bon moment, pas de state mort
   540→4. Flux de données bout en bout : tracer scénario complet clic→API→store→render
   541→5. Communications entre services : ports/URLs, XTransformPort, WebSockets, timeouts
   542→6. Références croisées entre livrables (documents)
   543→7. Corriger chaque problème d'interaction
   544→ÉTAPE 5 — Cohérence des raisonnements et résultats :
   545→1. Cohérence logique (pas de saut, pas de contradiction)
   546→2. Cohérence numérique (chiffres, pourcentages, tailles)
   547→3. Cohérence temporelle (dates, versions, chronologies)
   548→4. Résultat attendu vs résultat obtenu
   549→5. Cohérence entre fichiers livrables
   550→6. Corriger toute incohérence
   551→DÉPENDANCE : Ce skill dépend de gen-plan pour son Étape 1. Si gen-plan n'est pas disponible, l'Étape 1 doit être réalisée manuellement.

   727→## 8. RÈGLES D'OR (7 règles)
   728→
   729→1. **Règle 1** : Toujours lire le projet/système avant de planifier. Cette règle fondamentale garantit que toute planification ou action est basée sur une compréhension complète du contexte existant, évitant les erreurs d'assomption et les actions inutiles ou redondantes.
   730→
   731→2. **Règle 2** : Sélectionner l'agent optimal selon la performance. Utiliser la matrice de décision performance-driven pour choisir le meilleur agent pour chaque tâche, en privilégiant toujours la combinaison skill + agent spécialisé pour un résultat optimal.
   732→
   733→3. **Règle 3** : Exécuter en mode sériel par défaut (une tâche à la fois). L'exécution séquentielle garantit la qualité et la traçabilité de chaque étape, permettant de vérifier les outputs avant de passer à la tâche suivante.
   734→
   735→4. **Règle 4** : Vérifier les outputs avant de continuer. Chaque résultat produit doit être validé avant de passer à l'étape suivante, garantissant la qualité et la cohérence de l'ensemble du processus.
   736→
   737→5. **Règle 5** : Logger chaque phase dans worklog. La traçabilité est essentielle : chaque action, décision et résultat doit être consigné dans le worklog partagé pour permettre le suivi et la coordination entre agents.
   738→
   739→6. **Règle 6** : Communiquer clairement en français. Toute communication avec François doit être en français, conformément à sa préférence linguistique exprimée dans son profil.
   740→
   741→7. **Règle 7** : Utiliser les 12 agents spécialisés quand pertinent. L'écosystème complet d'agents spécialisés doit être exploité au maximum pour tirer parti de l'expertise spécifique de chacun, plutôt que de se limiter aux agents génériques.
