    34→    34→## 2. PHILOSOPHIE DE L'ÉCOSYSTÈME
    35→    35→
    36→    36→### Principe Fondamental
    37→    37→**Performance-driven selection** : Le choix entre skill, agent spécialisé ou agent général est dicté par le gain de performance, pas par une hiérarchie rigide.
    38→    38→
    39→    39→### Modèle à Deux Couches
    40→    40→```
    41→    41→SKILL (protocole + connaissance domaine)
    42→    42→    ↓
    43→    43→AGENT SPÉCIALISÉ (exécution + outils)
    44→    44→```
    45→    45→
    46→    46→- Le **skill** fournit le "comment" (protocole, templates, critères qualité)
    47→    47→- L'**agent** fournit le "qui" (capacités spécialisées, outils, vitesse)
    48→    48→
    49→    49→### Principes Directeurs
    50→    50→1. **Read before planning** — Toujours lire le projet avant de planifier
    51→    51→2. **Performance-driven** — Sélectionner l'agent optimal pour chaque sous-tâche
    52→    52→3. **Serial by default** — Exécution sérielle obligatoire sauf exception
    53→    53→4. **Visible progress** — L'utilisateur sait toujours où on en est
    54→    54→5. **Quality gates** — Validation après chaque phase
    55→    55→
    56→    56→### Exécution Sérielle par Défaut
    57→    57→- **Toutes les tâches s'exécutent UNE À LA UNE** dans l'ordre défini
    58→    58→- **Pas d'agents simultanés** sauf demande explicite + indépendance prouvée
    59→    59→- **Quality gate entre phases** : vérifier les outputs avant de continuer
    60→    60→- **Corrections en vol** : réévaluer si l'état du projet change significativement

   285→   285→## 4. SKILL GEN-PLAN — Orchestration (v1.2.0)
   286→   286→
   287→   287→**Déclencheurs** : `gen-plan:`, `plan d'actions`, `orchestre`
   288→   288→
   289→   289→### 4.1 Modes
   290→   290→
   291→   291→| Mode | Déclencheur | Comportement |
   292→   292→|------|-------------|--------------|
   293→   293→| **TÂCHE** | `gen-plan: <description>` | Planifie et exécute une tâche spécifique |
   294→   294→| **PROJET** | `gen-plan:correct-work(projet)` | Analyse complète, corrections, mise à jour prompt-master |
   295→   295→| **CIBLE** | `gen-plan:correct-work(<cible>)` | Vérification/correction d'un élément spécifique |
   296→   296→
   297→   297→### 4.2 Protocole — 13 Étapes Séquentielles
   298→   298→
   299→   299→#### Étape 1 — Collecte et Analyse des Demandes
   300→   300→**Objectif** : Relire toute la conversation pour extraire chaque demande explicite et implicite.
   301→   301→- Relire tous les messages utilisateur (demandes initiales, specs, contraintes, corrections)
   302→   302→- Identifier demandes explicites (ce que l'utilisateur a directement demandé)
   303→   303→- Identifier demandes implicites (prérequis, effets de bord, conséquences logiques)
   304→   304→- Lister chaque demande sans ambiguïté
   305→   305→- **Méthode** : Lire conversation complète. Utiliser worklog.md si disponible.
   306→   306→
   307→   307→#### Étape 2 — Lecture et Analyse du Projet
   308→   308→**Objectif** : Traverser tous les fichiers du projet pour comprendre structure, architecture, dépendances, état actuel.
   309→   309→- **Principe** : Un plan sans connaissance du projet est générique et probablement inadéquat.
   310→   310→- Structure fichiers : arborescence, configs (package.json, tsconfig, etc.)
   311→   311→- Code source : chaque fichier significatif
   312→   312→- Schéma base de données (si applicable)
   313→   313→- Dépendances : imports inter-modules, dépendances externes
   314→   314→- Configuration : fichiers config pertinents
   315→   315→- Documentation : README, worklog, specs, fichiers référence
   316→   316→- **Méthode** : Utiliser Read, Glob, Grep pour traverser chaque fichier significatif.
   317→   317→
   318→   318→#### Étape 3 — Identification de la Nature du Projet
   319→   319→**Objectif** : Déterminer type projet, technologies, architecture, complexité.
   320→   320→- Type projet : web app, API, script, document, analyse, skill, etc.
   321→   321→- Technologies : framework, langage, base de données, build tools
   322→   322→- Architecture : monolithe, micro-services, fullstack, frontend only, backend only
   323→   323→- Complexité : nombre fichiers, profondeur arborescence, dépendances
   324→   324→
   325→   325→#### Étape 4 — Identification des Objectifs
   326→   326→**Objectif** : Lister chaque objectif explicitement, sans ambiguïté ni omission.
   327→   327→- Lister objectifs dérivés des demandes utilisateur
   328→   328→- Lister objectifs implicites (prérequis, effets de bord, contraintes)
   329→   329→- Vérifier que chaque objectif est mesurable (atteint ou non)
   330→   330→- Supprimer doublons
   331→   331→
   332→   332→#### Étape 5 — Décomposition en Sous-tâches Adaptées
   333→   333→**Objectif** : Décomposer chaque objectif en sous-tâches atomiques avec ordre logique d'exécution.
   334→   334→- Pour chaque objectif, identifier sous-tâches atomiques
   335→   335→- Ordonner logiquement (prérequis d'abord, puis exécution, puis validation)
   336→   336→- Adapter au type projet :
   337→   337→  - **Fullstack** : inclure vérification API frontend-backend, schéma DB, auth
   338→   338→  - **Frontend only** : inclure vérification responsive, accessibilité, composants
   339→   339→  - **Backend/API** : inclure vérification endpoints, validation, sécurité
   340→   340→  - **Document/PDF** : inclure vérification contenu, layout, cohérence données
   341→   341→  - **Script/automation** : inclure cas limites, robustesse, gestion erreurs
   342→   342→
   343→   343→#### Étape 6 — Détection des Dépendances
   344→   344→**Objectif** : Identifier dépendances entre sous-tâches et contraintes de précédence.
   345→   345→- Dépendances séquentielles (A doit finir avant B)
   346→   346→- Dépendances parallèles (A et B peuvent tourner simultanément)
   347→   347→- Dépendances conditionnelles (B tourne seulement si A réussit/échoue)
   348→   348→- Vérifier absence de dépendances circulaires
   349→   349→- **Méthode** : Construire graphe de dépendances. Identifier chemins critiques.
   350→   350→
   351→   351→#### Étape 7 — Priorisation
   352→   352→**Objectif** : Classer sous-tâches par impact sur résultat final.
   353→   353→
   354→   354→| Priorité | Définition |
   355→   355→|----------|------------||
   356→   356→| **Critique** | Échec invalide le résultat entier |
   357→   357→| **Importante** | Impact significatif sur qualité |
   358→   358→| **Secondaire** | Améliore résultat mais non bloquant |
   359→   359→
   360→   360→#### Étape 8 — Estimation des Risques
   361→   361→**Objectif** : Pour chaque sous-tâche, identifier risques potentiels et plans de fallback.
   362→   362→- Complexité technique : connaissance spécialisée requise ?
   363→   363→- Ambiguïté : clairement définie ou ouverte à interprétation ?
   364→   364→- Dépendance externe : repose sur API, service, outil qui pourrait échouer ?
   365→   365→- Fallback : que faire si sous-tâche échoue ? Contourner, reporter, simplifier ?
   366→   366→
   367→   367→#### Étape 9 — Structuration du Plan
   368→   368→**Objectif** : Produire un plan d'actions formel, complet, adapté au projet.
   369→   369→
   370→   370→Le plan doit inclure :
   371→   371→- **En-tête** : nature projet, objectifs, contraintes
   372→   372→- **Liste ordonnée d'étapes** avec IDs (1, 2-a, 2-b, 3...)
   373→   373→  - Pour chaque étape : objectif, fichiers concernés, dépendances, priorité
   374→   374→  - Indication parallélisme où applicable
   375→   375→  - Critères de validation par étape
   376→   376→- **Carte dépendances** : résumé relations étapes
   377→   377→- **Matrice risques** : risques et plans fallback
   378→   378→
   379→   379→#### Étape 10 — Validation du Plan
   380→   380→**Objectif** : Checklist :
   381→   381→- [ ] Le plan couvre-t-il toutes les demandes utilisateur ?
   382→   382→- [ ] Y a-t-il des étapes manquantes ?
   383→   383→- [ ] L'ordre d'exécution est-il logique (aucune dépendance violée) ?
   384→   384→- [ ] Le plan est-il adapté au projet (pas générique) ?
   385→   385→- [ ] Chaque étape a-t-elle un critère de validation clair ?
   386→   386→- [ ] Les risques principaux sont-ils couverts par des plans fallback ?
   387→   387→
   388→   388→Si problème trouvé, retourner à l'étape pertinente et corriger.

   390→   390→#### Étape 11 — Test et Correction Pré-Intégration
   391→   391→**Objectif** : Tester chaque fichier généré ou modifié AVANT de l'intégrer à l'éco-système.
   392→   392→

   395→   395→   - **Skill** : Fichier avec YAML frontmatter (name, description) ou contenu définissant un protocole/routine réutilisable
   396→   396→   - **Éco-système** : Fichier de configuration, prompt maître, contexte, mémoire, agent
   397→   397→   - **Utilitaire** : Script, outil, fichier temporaire — ne pas intégrer
   398→   398→2. **Tester le fichier** :
   399→   399→   - **Skill (.md avec YAML)** : Vérifier YAML valide, contenu cohérent (> 200 chars), pas de placeholders
   400→   400→   - **Python (.py)** : Compiler avec compile() ou ast.parse() pour vérifier syntaxe
   401→   401→   - **Shell (.sh)** : Vérifier shebang, syntaxe avec bash -n, pas de commandes destructrices
   402→   402→   - **Markdown (.md)** : Vérifier structure, absence de contenu tronqué
   403→   403→   - **Configuration (.json, .yaml)** : Valider format, vérifier clés obligatoires
   404→   404→3. **Corriger les anomalies détectées** immédiatement
   405→   405→4. **Valider la cohérence entre fichiers** (références croisées valides)
   406→   406→

   522→   522→- CIBLE : correct_work(cible) — vérification ciblée sur un livrable spécifique, sans gestion du prompt-maître

   528→   528→1. Relire les spécifications initiales et vérifier chaque exigence satisfaite
   529→   529→2. Vérifier les données factuelles (noms, chemins, versions, tailles, counts)
   530→   530→3. Vérifier la cohérence linguistique (pas de mélange de langues incohérent)
   531→   531→4. Vérifier les fichiers de sortie (existent, lisibles, non vides)
   532→   532→5. Vérifier les dépendances (imports, chemins, références croisées)
   533→   533→6. Adapter la vérification au type de projet
   534→   534→7. Corriger chaque erreur ou omission identifiée
   535→   535→ÉTAPE 3 — Structure du code et conflits :
   536→   536→1. Imports circulaires
   537→   537→2. Conflits de noms (fonctions/classes/variables en double)
   538→   538→3. Variables non initialisées ou utilisées avant définition
   539→   539→4. Chemins en dur non portables
   540→   540→5. Gestion des erreurs (cas d'erreur traités ou échec silencieux ?)
   541→   541→6. Doublons de code à factoriser
   542→   542→7. Convention de nommage cohérente
   543→   543→8. Conflits logiques et conditions contradictoires — utiliser un TABLEAU DE KARNAUGH pour simplifier, détecter les contradictions, vérifier la couverture complète
   544→   544→9. Corriger chaque problème identifié
   545→   545→ÉTAPE 4 — Vérification des interactions :
   546→   546→Adaptée au type de projet (fullstack, frontend, backend, document, script) :
   547→   547→1. API frontend-backend : endpoints existent, params correspondent, réponses parsées, codes erreur gérés
   548→   548→2. Props inter-composants : types corrects, callbacks bons args, pas de props manquantes
   549→   549→3. State management : store expose données nécessaires, actions au bon moment, pas de state mort
   550→   550→4. Flux de données bout en bout : tracer scénario complet clic→API→store→render
   551→   551→5. Communications entre services : ports/URLs, XTransformPort, WebSockets, timeouts
   552→   552→6. Références croisées entre livrables (documents)
   553→   553→7. Corriger chaque problème d'interaction
   554→   554→ÉTAPE 5 — Cohérence des raisonnements et résultats :
   555→   555→1. Cohérence logique (pas de saut, pas de contradiction)
   556→   556→2. Cohérence numérique (chiffres, pourcentages, tailles)
   557→   557→3. Cohérence temporelle (dates, versions, chronologies)
   558→   558→4. Résultat attendu vs résultat obtenu
   559→   559→5. Cohérence entre fichiers livrables
   560→   560→6. Corriger toute incohérence

   738→   738→## 8. RÈGLES D'OR (7 règles)
   739→   739→
   740→   740→1. **Règle 1** : Toujours lire le projet/système avant de planifier. Cette règle fondamentale garantit que toute planification ou action est basée sur une compréhension complète du contexte existant, évitant les erreurs d'assomption et les actions inutiles ou redondantes.
   741→   741→
   742→   742→2. **Règle 2** : Sélectionner l'agent optimal selon la performance. Utiliser la matrice de décision performance-driven pour choisir le meilleur agent pour chaque tâche, en privilégiant toujours la combinaison skill + agent spécialisé pour un résultat optimal.
   743→   743→
   744→   744→3. **Règle 3** : Exécuter en mode sériel par défaut (une tâche à la fois). L'exécution séquentielle garantit la qualité et la traçabilité de chaque étape, permettant de vérifier les outputs avant de passer à la tâche suivante.
   745→   745→
   746→   746→4. **Règle 4** : Vérifier les outputs avant de continuer. Chaque résultat produit doit être validé avant de passer à l'étape suivante, garantissant la qualité et la cohérence de l'ensemble du processus.
   747→   747→
   748→   748→5. **Règle 5** : Logger chaque phase dans worklog. La traçabilité est essentielle : chaque action, décision et résultat doit être consigné dans le worklog partagé pour permettre le suivi et la coordination entre agents.
   749→   749→
   750→   750→6. **Règle 6** : Communiquer clairement en français. Toute communication avec François doit être en français, conformément à sa préférence linguistique exprimée dans son profil.
   751→   751→
   752→   752→7. **Règle 7** : Utiliser les 12 agents spécialisés quand pertinent. L'écosystème complet d'agents spécialisés doit être exploité au maximum pour tirer parti de l'expertise spécifique de chacun, plutôt que de se limiter aux agents génériques.
