    39→### ⚠️ Pre-Routing Checks (run BEFORE matching brief)
    40→
    41→1. **Emoji Check** - Scan user content for intentional emoji (decorative 📊🎯🔥, not OS-level emoji input). If found → **force Creative pipeline** (Fixed-Canvas or Flow depending on document type) regardless of original routing. ReportLab renders emoji as □ squares; LaTeX drops them entirely.
    42→2. **CJK Check** - Chinese/Japanese/Korean content needs font coverage. Report brief must register CJK fonts - **probe first** with `ls /usr/share/fonts/truetype/chinese/` (Linux) or check `$FONT_DIR` (macOS) to confirm which fonts exist, then register accordingly (prefer NotoSerifSC > Noto Sans SC; never hardcode a font name without verifying it exists). Creative Fixed-Canvas and Creative Flow briefs must load Google Fonts Noto Sans SC with `font-display: swap`; Academic brief must use `\usepackage{ctex}`.
    43→3. **Size Check** - Non-standard page sizes (not A4/Letter/A3) → prefer Creative brief (Playwright handles any dimension). ReportLab can do custom sizes but pagination is manual.
    44→4. **Character Safety Check** - Before writing any content string, scan for Japanese kana (の、が、は etc.), unusual Unicode symbols, or non-CJK characters that may corrupt during encoding transit ( Especially when code is written via heredoc/base64/LLM output). Replace with plain Chinese equivalents: `の`→`之/的/缔`, `々`→omit or write full character. **If content must preserve Japanese, use only standard CJK Unified Ideographs (U+4E00-U+9FFF) and common kana; avoid rare/private-use codepoints.**

   197→## Content Rules
   198→
   199→- **Language**: Match user's query language. Chinese query → Chinese PDF.
   200→- **Page/word count**: Respect explicit constraints (±20%). Unspecified → completeness over brevity.
   201→- **Outline**: User-provided outlines are sacred. No reordering without asking.
   202→- **Citations**: No fabrication. Chinese → GB/T 7714, English → APA. Search to verify.
   203→- **Multi-part requests**: Generate ALL parts - never silently drop a component.
   204→
   205→### Content Depth and Richness Standards (Anti-Shallow Writing)
   206→
   207→**PROBLEM TO AVOID**: Shallow content with paragraphs containing only 1-2 sentences, or sections with minimal text under headings.
   208→
   209→#### Minimum Content Standards
   210→
   211→1. **Paragraph Depth**
   212→   - Each paragraph MUST contain at least 3-5 sentences
   213→   - Single-sentence paragraphs are FORBIDDEN (except for transitional statements)
   214→   - Each paragraph should develop ONE complete idea with supporting details
   215→
   216→2. **Section Completeness**
   217→   - Each section heading MUST be followed by substantial content (minimum 150-200 words)
   218→   - NEVER create a section with only 1-2 short sentences
   219→   - If a section cannot meet minimum length, merge it with related sections
   220→
   221→3. **Content Enrichment Techniques**
   222→   - Include specific examples, data points, or case studies
   223→   - Provide context and background information
   224→   - Explain the "why" and "how", not just the "what"
   225→   - Add comparisons, contrasts, or alternative perspectives where relevant
   226→   - Include implications, consequences, or recommendations
   227→
   228→#### Writing Quality Checklist
   229→
   230→Before finalizing ANY written document, verify:
   231→- [ ] No paragraph has fewer than 3 sentences
   232→- [ ] No section has fewer than 150 words of body content
   233→- [ ] Each main point is supported by examples or evidence
   234→- [ ] Technical terms are explained when first introduced
   235→- [ ] Transitions connect ideas between paragraphs and sections
   236→
   237→#### Content Expansion Strategies
   238→
   239→When writing feels thin, apply these techniques:
   240→
   241→**For Analysis/Reports:**
   242→- Add background context (history, current state, trends)
   243→- Include multiple perspectives or stakeholder views
   244→- Provide specific metrics, statistics, or quantitative data
   245→- Discuss limitations, challenges, or counterarguments
   246→- Offer actionable recommendations with rationale
   247→
   248→**For Explanatory Content:**
   249→- Use analogies to clarify complex concepts
   250→- Provide step-by-step breakdowns where applicable
   251→- Include real-world applications or use cases
   252→- Address common questions or misconceptions
   253→- Add visual descriptions or diagram explanations
   254→
   255→**FORBIDDEN PATTERNS:**
   256→- Section with heading followed by only 1-3 sentences
   257→- Bullet lists without explanatory context
   258→- Conclusions that merely restate the introduction
   259→- Sections that say "as mentioned above" without adding new value
   260→
   261→### Language Consistency Rule
   262→
   263→Always use the same language as the user's input language for:
   264→- The response and document content
   265→- Any generated report, PDF, or plot/diagram
   266→- **Cover page**: All text on the cover (title, subtitle, tags, author, footer) MUST match the user's prompt language. Chinese prompt → Chinese cover; English prompt → English cover. Mixing is forbidden.
   267→- **Charts and figures**: Before generating any chart or figure, determine the user's language. Ensure that the title, legend, labels, and other textual elements are consistent with the user's language. If any element cannot use the user's language, explicitly explain the reason.

   269→### HTML Image Source Path Rules
   270→
   271→When embedding images in HTML documents (Creative pipeline, Playwright-rendered diagrams, or any HTML→PDF flow):
   272→
   273→| Image location | `<img src>` value | Example |
   274→|---|---|---|
   275→| **Local file** | **Relative path** from the HTML file's directory | `<img src="images/chart.png">` or `<img src="./diagram.png">` |
   276→| **Remote URL** | Full URL (no change needed) | `<img src="https://example.com/photo.jpg">` |
   277→
   278→**Iron rules:**
   279→1. **NEVER use absolute paths** for local files in HTML `<img>`, `<source>`, CSS `url()`, or any other asset reference (e.g. `/Users/alice/project/img.png`). Absolute paths break portability across machines and environments.
   280→2. **Always use relative paths** anchored to the HTML file's own directory. If the image lives in a subdirectory, use `images/foo.png` or `./images/foo.png`.
   281→3. **Remote URLs (`http://` / `https://`) are fine as-is** - do not convert them to local paths.
   282→4. When generating HTML from a script or blueprint, ensure all referenced assets are either (a) in the same directory as the output HTML, or (b) in a clearly named subdirectory (e.g. `assets/`, `images/`), and referenced with relative paths.
   283→5. If a build script needs to resolve paths programmatically, compute relative paths at generation time (e.g. `os.path.relpath(image_path, html_dir)`) rather than embedding absolute filesystem paths.

   287→## Figure & Diagram Embedding (All Briefs)
   288→
   289→### Iron Rule: Figures Are Block-Level
   290→
   291→Figures, diagrams, and charts MUST be independent block elements occupying full width. **Never** float/wrap figures alongside body text - this causes the text-diagram overlap badcase.
   292→
   293→| Brief | Correct embedding | Forbidden |
   294→|-------|-------------------|-----------|
   295→| Report (ReportLab) | `story.append(Image(...))` as standalone Flowable | Placing images inside Paragraph text, simulating float |
   296→| Creative (Playwright) | `<figure style="display:block; width:100%; margin:2em auto">` | `float:right`, `display:flex` with text, `wrapfigure`-style CSS |
   297→| Academic (LaTeX) | `\begin{figure}[t] ... \end{figure}` | Bare `\includegraphics` in text body (no figure env), bare `tikzpicture` in multi-column |
   298→
   299→### Complex Diagram Strategy
   300→
   301→When a diagram has **>12 nodes, >3 subgroups, or intricate connections**, do NOT try to render it as one giant figure. Instead:
   302→
   303→1. **Table for details** - structured data (phases, components, specs) goes into a proper table
   304→2. **Simplified overview diagram** - a stripped-down flowchart/Mermaid showing only the top-level flow (≤8 nodes)
   305→3. **Cross-reference** - table caption + diagram caption reference each other
   306→
   307→This "table + simple diagram" pattern prevents:
   308→- Diagrams overflowing page boundaries
   309→- Text becoming unreadably small to fit everything
   310→- Layout engines mishandling oversized graphics
   311→
   312→### Diagram Content Quality Rules (Cross-reference: charts)
   313→
   314→The rules above handle **how** to embed diagrams in PDF. For **what the diagram itself looks like** (node layout, connector routing, color, readability), follow the `charts` skill rules:
   315→
   316→**Before generating ANY flowchart/diagram for PDF embedding, check these:**
   317→
   318→1. **Connectors must not pass through nodes** - If 3+ layers exist, connect adjacent layers only (top→mid, mid→bottom). Never draw top→bottom lines through middle nodes. Use detour paths if cross-layer links are needed.
   319→2. **Multiple arrows into one node must not pile up** - Distribute entry points evenly along target edge, or use merge-then-enter pattern (sources converge to a vertical merge line, then single arrow to target).
   320→3. **Low-saturation fills only** - Node backgrounds must be pale (`#EFF6FF`, `#F0FDF4`). High-saturation colors (`#3B82F6`, `#10B981`) only for borders or small accents. No children's-art color schemes.
   321→4. **Phase titles vs sub-steps must be visually distinct** - Different background color, font size, and font weight. Never same-style boxes for both.
   322→5. **Font sizes must be readable at final output size** - Sizes depend on the embedding context:
   323→   | Output context | Node title min | Description min | Label min |
   324→   |---------------|----------------|-----------------|-----------|
   325→   | Standalone PNG (web/presentation, ≥1200px wide) | 14px | 12px | 11px |
   326→   | Embedded in A4 PDF (ReportLab/LaTeX, ~450pt content width) | 10pt | 8pt | 7pt |
   327→   | Embedded in slide deck (landscape, ~720pt wide) | 12pt | 10pt | 9pt |
   328→
   329→   **Principle**: After embedding, the smallest text in the diagram must still be legible when the document is viewed at 100% zoom. If the diagram is scaled down to fit page width, recalculate: `effective_size = original_size × (display_width / canvas_width)`. If effective size drops below the minimum, either increase original font size or reduce diagram complexity.
   330→6. **Legend/annotations must not overlap content** - Separate container, ≥ 40px gap from last node, fully within canvas bounds.
   331→
   332→**For Playwright-rendered diagrams**: Use low-saturation fills (`#EFF6FF`, `#F0FDF4`), CSS flexbox/grid for node layout, SVG `<line>`/`<path>` for connectors, and verify no overlap at final render size.
   333→**For ReportLab-drawn diagrams**: Same principles apply - use `Drawing()` with explicit coordinates, check node bounding boxes for overlap before finalizing.
   334→
   335→### Diagram Generation Strategy (Per-Brief)
   336→
   337→Diagram rendering depends on the target brief - **NOT** a one-size-fits-all TikZ pipeline.
   338→
   339→| Target Brief | Diagram Method | Rationale |
   340→|---|---|---|
   341→| **Report** (ReportLab) | Playwright+CSS → PNG → `Image()` | No LaTeX compiler in this route; HTML/CSS handles any layout natively |
   342→| **Creative** (Playwright) | Directly in HTML (CSS flexbox/grid + JS connectors) | Already in browser context |
   343→| **Academic** (Tectonic) - simple (≤6 nodes) | TikZ native `tikzpicture` | Vector output, font consistency, LaTeX-native |
   344→| **Academic** (Tectonic) - complex (>6 nodes) | Playwright+CSS → PNG @2× → `\includegraphics` | TikZ branch logic is error-prone for models; 300dpi PNG is publication-ready |
   345→
   346→**Playwright+CSS diagram pipeline (Report & Academic-complex):**
   347→
   348→```bash
   349→# 1. Write diagram HTML (CSS grid/flexbox + connectors)

   379→## Vector Rendering Iron Rule
   380→
   381→**The final PDF MUST be generated via `page.pdf()` (Playwright) or ReportLab/LaTeX native output - NEVER via screenshot-to-PDF.**
   382→
   383→| Scenario | Correct Method | Forbidden |
   384→|----------|---------------|-----------|
   385→| Creative pipeline (single/multi-page) | `page.pdf()` via `convert.blueprint` or `html2pdf-next.js` | `page.screenshot()` → image → wrap as PDF |
   386→| Report cover (HTML/Playwright) | `page.pdf()` → merge via pypdf | Screenshot cover → embed as image |
   387→| Academic cover | `page.pdf()` → merge via pypdf | Screenshot → `\includegraphics` for cover |
   388→| Full-page posters/infographics | `html2poster.js` (auto overflow:hidden + height measurement + `page.pdf()`) | Any raster pipeline for the final output |
   389→
   390→**Why:** `page.pdf()` produces vector text + vector shapes. Text remains selectable, sharp at any zoom, and file size is smaller. Screenshot-based PDFs are raster images - blurry when zoomed, unsearchable, and 3-5× larger.
   391→
   392→**The ONLY place screenshot/PNG embedding is acceptable:**
   393→- **Diagrams** embedded as sub-elements inside a larger document (e.g., flowcharts in a Report). These use `page.screenshot()` at 2× device scale factor for 300dpi print quality, then embed via `Image()` (ReportLab) or `\includegraphics` (LaTeX).
   394→- **Chart images** generated by matplotlib/plotly saved as PNG, then embedded.
   395→
   396→These are sub-elements, not the document itself. The document-level PDF output must always be vector.
   397→
   398→**Quick test:** Open the generated PDF, zoom to 400%. If text is blurry, you used a screenshot pipeline. Fix it.
   399→
   400→### HTML→PDF Engine Selection Rules
   401→
   402→There are **two dedicated scripts** for HTML→PDF. Choose based on document type:
   403→
   404→| Document type | Script | Reason |
   405→|---------------|--------|--------|
   406→| **Posters, infographics, long-image single-page designs** | `html2poster.js` | Auto overflow:hidden, auto height measurement, zero margin, single-page output |
   407→| **Cover pages (Report/Academic route)** | `html2poster.js` | Covers are single-page fixed layouts with absolute positioning - same nature as posters. `html2pdf-next.js` would convert absolute→static and destroy the layout |
   408→| **Multi-page documents, reports, academic papers, resumes** | `html2pdf-next.js` | Paged.js pagination, A4/custom size, pdf-lib metadata. `--nopaged` for Chromium native fallback |
   409→| **Creative pipeline (Blueprint → HTML → PDF)** | `html2pdf-next.js` via `convert.blueprint` | Called internally by design_engine pipeline |
   410→
   411→#### Poster / Single-Page Long-Image → `html2poster.js`
   412→
   413→```bash
   414→node "$PDF_SKILL_DIR/scripts/html2poster.js" poster.html --output poster.pdf --width 720px
   415→```
   416→
   417→`html2poster.js` automatically:
   418→- Forces `overflow: hidden` on `.poster` / `.page` containers (clips decorative overflow)
   419→- Injects `@page { margin: 0 }` (zero margins always)
   420→- Syncs `html/body` background with poster background color
   421→- Measures `.poster` scrollHeight and uses it as PDF height
   422→- Generates a single-page vector PDF with exact content dimensions
   423→
   424→**Use this for ANY fixed-width, dynamic-height, single-page design.**
   425→
   426→#### Documents / Multi-Page → `html2pdf-next.js`
   427→
   428→```bash
   429→node "$PDF_SKILL_DIR/scripts/html2pdf-next.js" input.html --output output.pdf --width 210mm --height 297mm
   430→# Or via pdf.py wrapper:
   431→python3 "$PDF_SKILL_DIR/scripts/pdf.py" convert.html input.html --output output.pdf
   432→```
   433→
   434→Pre-render hooks auto-handle Mermaid/KaTeX rendering, overflow detection, and font loading. **Paged.js polyfill** is injected for pagination (break-inside/before/after, orphans/widows, named pages). Use `--nopaged` flag to fall back to Chromium native @page pagination if needed.
   435→
   436→#### ⚠️ Iron Rule: No Hand-Written Playwright Scripts
   437→
   438→Common issues with hand-written Python `page.pdf()` (the dedicated scripts handle these automatically):
   439→1. **Missing `@page` rule** → browser default margin causes content overflow to second page or white edges
   440→2. **Oversized elements not fixed** → large elements with `break-inside: avoid` block pagination, content gets truncated
   441→3. **Rendering before fonts are loaded** → Chinese text displays as squares or falls back to wrong font
   442→4. **No overflow detection** → content exceeds page boundary without awareness
   443→5. **No metadata** → PDF title, author, and other info missing
   444→
   445→**Iron rule: Posters and cover pages use `html2poster.js`, multi-page documents use `html2pdf-next.js`. Do not write hand-written Python Playwright scripts.**
   446→
   447→> **⚠️ Cover page gotcha:** Cover HTML uses `position: absolute` for layout. Always use `html2poster.js` for cover pages — `html2pdf-next.js` + Paged.js would re-layout absolute-positioned elements into flow, destroying the cover design.
   448→
   449→### No overflow:hidden on Fixed-Size Pages
   450→
   451→**NEVER set `overflow: hidden` on `html`, `body`, `.page`, or the main container** in HTML intended for PDF conversion. Paged.js handles content chunking and pagination natively — it does not need overflow clipping.
   452→
   453→> **Note:** This rule does NOT apply to posters rendered via `html2poster.js` - that script automatically adds `overflow: hidden` to `.poster`/`.page` containers to clip decorative overflow.
   454→
   455→| Problem | Cause | Fix |
   456→|---------|-------|-----|
   457→| Content silently clipped at page boundaries | `overflow: hidden` on container hides content exceeding bounds | Remove `overflow: hidden`; Paged.js handles pagination |
   458→| Decorative elements inflate scrollWidth | `width > 100%` or negative offsets on absolutely-positioned elements | Constrain within page bounds (see creative-fixed-canvas.md §0.75) |
   459→
   460→**Always pair fixed-size pages with `@media screen` auto-scale** so the full page is visible in any browser window without scrolling. See `briefs/creative-fixed-canvas.md` § 0.5 for the CSS pattern.
   461→
   462→### Full-Bleed Rule (No White Margins)
   463→
   464→When generating HTML for Playwright `page.pdf()`, the content **MUST fill the entire page** with zero margins. White side margins = broken layout.
   465→
   466→**Mandatory CSS for any HTML → PDF:**
   467→```css
   468→@page {
   469→  size: <width> <height>;  /* e.g., 720px 960px, or A4 */
   470→  margin: 0;
   471→}
   472→html, body {
   473→  margin: 0;
   474→  padding: 0;
   475→}
   476→```
   477→
   478→**Common causes of white margins:**
   479→1. Missing `@page { margin: 0 }` - browser default margins kick in (~1cm each side)
   480→2. Content width doesn't match page width - e.g., canvas is 720px but page is A4 (794px)
   481→3. Missing `@page { size }` declaration in the HTML
   482→4. Content has explicit `max-width` that's narrower than the page
   483→
   484→**For blueprint pipeline:** `design_engine.py` now injects `@page { size: var(--canvas-w) var(--canvas-h); margin: 0; }` automatically.
   485→**For raw HTML:** YOU must include the `@page` rule. No exceptions.
   486→**For direct Playwright:** Pass `margin: { top: 0, right: 0, bottom: 0, left: 0 }` to `page.pdf()`.
   487→
   488→### Background Color Consistency (No Color Mismatch)
   489→
   490→**`html` / `body` background color must match the content canvas background color.**
   491→
   492→Playwright `page.pdf({ printBackground: true })` renders the body background color. If body is white while the content area is gray/colored, color-inconsistent borders/gaps will appear in the PDF.
   493→
   494→#### Single-color documents (all pages same background)
   495→
   496→```css
   497→/* MANDATORY: body background = content background */
   498→html, body {
   499→  margin: 0;
   500→  padding: 0;
   501→  background: var(--c-bg);  /* Same color as content canvas */
   502→}
   503→```
   504→
   505→#### Multi-page documents with mixed backgrounds (e.g. dark cover + white body pages)
   506→
   507→**Root cause:** Playwright resolves `.page { width: 210mm }` and `@page { size: 210mm }` to slightly different sub-pixel values (e.g. 793.688px vs 793.701px). This creates a <1px gap at the right/bottom edge of each `.page` div where `body`'s background shows through. On dark pages, a white `body` background makes this gap visible as a white edge.
   508→
   509→**Fix - set `body` background to the document's dominant dark color:**
   510→
   511→```css
   512→:root {
   513→  --primary: #0f172a;  /* darkest page background */
   514→}
   515→html, body {
   516→  margin: 0;
   517→  padding: 0;
   518→  width: 210mm;  /* match @page size */
   519→  background: var(--primary);  /* fallback for sub-pixel gaps */
   520→}
   521→```
   522→
   523→**Why this works and doesn't break white pages:**
   524→- Dark pages: sub-pixel gap reveals dark `body` → gap invisible.
   525→- White pages: `.page-white { background: #ffffff }` fully covers `body` → dark body never visible.
   526→- The gap is <1px - even on white pages, the dark body at the extreme pixel edge is imperceptible after anti-aliasing.
   527→
   528→**Rule: when generating multi-page HTML with mixed backgrounds, always set `html, body { background }` to the darkest page's background color.** If all pages are light/white, use the lightest content background (e.g. `#f8fafc`). Never leave `body` background unset (browser default = white = guaranteed white edges on dark pages).
   529→```
   530→
   531→### Content Centering (No Left/Right Drift)
   532→
   533→**After HTML-to-PDF conversion, content must be centered, no left or right drift allowed.**
   534→
   535→Common drift causes:
   536→1. `@page { margin }` not 0 - browser default margin causes drift
   537→2. `.safe-zone` or content container `inset` / `padding` left-right asymmetric
   538→3. Content container has `max-width` but no `margin: 0 auto`
   539→4. Grid components only occupy partial column width (e.g. `1/1 → X/7` only uses left half)
   540→5. **Decorative elements overflow page boundary** - elements with `width > 100%` or negative offsets (e.g. glow circles, gradient overlays) inflate `scrollWidth` beyond page width. Playwright shrinks all content to fit, causing left-shift. **Fix: constrain decorative elements within page bounds** (`width` ≤ 100%, no negative `left`/`right` offsets). See `briefs/creative-fixed-canvas.md` §0.75 and `typesetting/overflow.md` §3.5 for details.
   541→
   542→### Anti-Void Edges (No Large Blank Margins)
   543→
   544→**Content should not have large meaningless whitespace at page edges, top, or bottom.**
   545→
   546→- Content should make full use of page area; do not cram all content in the top half while leaving the bottom blank
   547→- For multi-page documents, each page's fill rate should be ≥ 60% (see `pagination.md` last page ≥ 40% rule)
   548→- For single-page posters/infographics, fill rate should be ≥ 70%

   580→### Overflow Prevention System
   581→
   582→**→ Full spec: `typesetting/overflow.md`** - read it for any document with tables, images, or multi-column layouts.
   583→
   584→Core principles:
   585→1. **Measure first, draw second** - never render content without pre-calculating its dimensions
   586→2. **Bounding Box constraint** - every element's width ≤ its parent container's `Max_Width`
   587→3. **Text: use font metrics**, not character count, for width calculation
   588→4. **Images: proportional scaling** - never insert at original size
   589→5. **Tables: weight-based column width** + `Paragraph()` wrapping (never plain strings)
   590→6. **Fallback ladder**: wrap → shrink font (max -3pt) → reduce padding → split element → log warning
   591→7. **Vertical: KeepTogether** for heading+body, chart+caption; `repeatRows=1` for long tables
   592→
   593→### Table Overflow Prevention (ReportLab)
   594→**Most common layout bug: table columns exceed page margins.**
   595→
   596→Before building any ReportLab Table:
   597→1. Calculate `available_width = page_width - left_margin - right_margin`
   598→2. Use proportional colWidths (`[0.25, 0.40, 0.20, 0.15]` × available_width) or fixed+flex pattern
   599→3. `sum(colWidths)` must be ≤ `available_width` - **verify this in code**
   600→4. Long text columns must use `Paragraph()` wrapping, not plain strings (plain strings don't wrap)
   601→5. CJK text is wider: budget ~12pt per character at 10pt font size
   602→
   603→See `briefs/report.md` § "Table Width Management" for code patterns.
   604→
   605→### Table Overflow Prevention (LaTeX/Academic)
   606→**Most common bug in dual-column papers: wide tables overflow single-column width.**
   607→
   608→Before writing any LaTeX table:
   609→1. Count data columns - ≤ 4 fits single column; 5-6 needs `\small`; 7-8 needs `\resizebox`; ≥ 9 use `table*` (full width)
   610→2. Use `tabular*{\columnwidth}` or `tabularx{\columnwidth}` instead of plain `tabular` for 5+ columns
   611→3. Never use plain `tabular` with 8+ columns in twocolumn layout - guaranteed overflow
   612→4. `\resizebox{\columnwidth}{!}` as last resort - verify smallest text ≥ 6pt after scaling
   613→
   614→See `briefs/academic.md` § "Table width management" for LaTeX patterns.
   615→
   616→### Playwright PDF CSS Blacklist
   617→These CSS properties **silently break** in Playwright's PDF renderer:
   618→- `backdrop-filter` / `-webkit-backdrop-filter` - **drops entire element content**. Use solid `rgba()` backgrounds.
   619→- `overflow: hidden` on content containers - clips content. Only safe on small decorative elements (< 200px).
   620→
   621→After generating any Playwright PDF, **verify every page has content** (pypdf text extraction, check non-empty).
   622→

   624→ALL PDFs must have: Title, Author (default "Z.ai"), Creator, Subject.

   828→## 8. Quality Checklist (Mandatory after every PDF generation)
   829→
   830→> ⚠️ **正文才是第一章。** 封面/目录/摘要不计入编号，正文编号永远从 1 开始。详见 `report.md` Step 3.5。
   831→
   832→> The following checks come from the `typesetting/` spec files and are **mandatory** quality gates.
   833→
   834→### Automated Detection (Must Run)
   835→
   836→```bash
   837→# 1. PDF quality check (all pipelines)
   838→python3 "$PDF_SKILL_DIR/scripts/pdf_qa.py" <output.pdf>
   839→python3 "$PDF_SKILL_DIR/scripts/pdf_qa.py" --poster <output.pdf>   # poster mode: skip content fill ratio, check all pages for full-bleed
   840→python3 "$PDF_SKILL_DIR/scripts/pdf_qa.py" --skip-cover --formulas <output.pdf>   # academic mode: skip cover for margin check, enable formula overflow
   841→python3 "$PDF_SKILL_DIR/scripts/pdf_qa.py" --no-tables <output.pdf>   # creative mode: skip table centering check
   842→
   843→# 2. Cover overlap detection (Report/Academic with cover - MANDATORY)
   844→#    Run on the cover HTML BEFORE rendering to PDF. Detects:
   845→#    - Text vs decorative line overlap (minimum gap = 1U = 5% of page width)
   846→#    - Layer 3 text vs text zone overflow (same-layer foreground text blocks overlapping)
   847→#    Exit 0 = pass, Exit 1 = overlap found (must fix), Exit 2 = script error
   848→node "$PDF_SKILL_DIR/scripts/cover_validate.js" cover.html
   849→node "$PDF_SKILL_DIR/scripts/cover_validate.js" cover.html --min-gap 30   # custom min gap in px
   850→```
   851→
   852→> **Dependencies**: `pymupdf` (`pip install pymupdf`) for pdf_qa.py; `playwright` or `playwright-core` for cover_validate.js. If not installed, skip the respective check and use the manual checklist below.
   853→
   854→Run `pdf_qa.py` after generating a PDF. It auto-detects: metadata completeness, page size consistency, blank pages, CJK punctuation placement, color count, font embedding status, content overflow, content fill ratio, cover full-bleed, margin symmetry, table centering, formula overflow.
   855→- **`--poster` mode**: skips content fill ratio check (poster last page naturally has less content), checks ALL pages for full-bleed (not just cover)
   856→- **`--skip-cover`**: skips page 1 when checking margin symmetry (for documents with separately-generated covers)
   857→- **`--no-tables`**: disables table centering check (for creative/poster documents that rarely have traditional tables)
   858→- **`--formulas`**: enables formula overflow detection (checks if formula-like content extends past right content margin)
   859→- Result PASS → deliver directly
   860→- Result WARN → evaluate whether fix is needed, non-blocking
   861→- Result FAIL → **must fix and regenerate**
