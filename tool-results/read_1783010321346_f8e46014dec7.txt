    28→## 1. Identité & Philosophie
    29→
    30→### 1.1 Définition
    31→
    32→**Prompt-Engineering-Kit v4.0 (PEK v4.0)** est une application full-stack de génération automatisée de kits de prompt engineering. Elle transforme tout projet IA en un kit structuré, réutilisable et validé, via une méthodologie **SUPER-AGENT HYBRIDE CoT + CHAINING**.
    33→
    34→### 1.2 Principe Fondamental
    35→
    36→L'application ne génère pas un simple prompt — elle **décompose un projet source en blocs modulaires normalisés (A-J)**, les traite séquentiellement via un **pipeline Chain-of-Thought à 7 étapes**, et produit un **kit complet, scoré et exportable**.
    37→
    38→### 1.3 Méthodologie : SUPER-AGENT HYBRIDE CoT + CHAINING
    39→
    40→- **CoT (Chain-of-Thought)** : Chaque étape du pipeline raisonne logiquement sur la précédente. Les sorties s'accumulent et se nourrissent mutuellement.
    41→- **Chaining** : Les 7 étapes s'exécutent de manière **strictement séquentielle**. L'étape N+1 reçoit toujours les sorties des étapes 1 à N.
    42→- **Hybride** : Combine le raisonnement LLM (génération de contenu) avec des mécanismes déterministes (validation par règles, scoring algorithmique).
    43→- **Super-Agent** : Le système agit comme un architecte expert autonome qui orchestre l'ensemble du processus.
    44→
    45→### 1.4 Langue
    46→
    47→L'application est **en français** par défaut (`lang="fr"` sur `<html>`). Les contenus générés sont en français sauf configuration contraire (champ `language: "en"`).

   214→### 3.2 Relations Clés
   215→
   216→- **Project 1→N Session** : Un projet contient plusieurs sessions. Suppression en cascade.
   217→- **Session 1→N Step** : Chaque session a exactement 7 étapes (créées à l'initialisation).
   218→- **Session 1→N Validation** : Chaque session peut avoir plusieurs validations (historique).
   219→- **Step appartient à Session** : Cascade delete.

   320→## 5. Pipeline Chain-of-Thought — Spécification Complète
   321→
   322→### 5.1 Vue d'Ensemble
   323→
   324→Le pipeline est le cœur de l'application. Il transforme une session (projet + configuration) en un kit structuré via 7 appels LLM séquentiels avec accumulation de contexte.
   325→
   326→```
   327→[Session Input] → Étape 1 → Étape 2 → ... → Étape 7 → [Kit Final + Score]
   328→                      ↑                    ↑
   329→                      └── Chaining : chaque étape reçoit les sorties des précédentes
   330→```
   331→
   332→### 5.2 Détail de Chaque Étape
   333→
   334→#### Étape 1 — COMPRÉHENSION
   335→
   336→**Objectif** : Analyse en profondeur du projet fourni.
   337→
   338→**Sorties attendues** :
   339→- Objectif principal du projet IA
   340→- Domaine d'application
   341→- Technologies et frameworks utilisés
   342→- Acteurs impliqués
   343→- Niveau de complexité global
   344→- Points clés à préserver
   345→
   346→**Format** : Analyse structurée avec sections claires.
   347→
   348→---
   349→
   350→#### Étape 2 — DÉCOMPOSITION
   351→
   352→**Objectif** : Découpage du projet en composants réutilisables.
   353→
   354→**Sorties attendues** :
   355→- Responsabilité principale de chaque composant
   356→- Entrées/sorties définies
   357→- Dépendances inter-composants
   358→- Évaluation de réutilisabilité
   359→- Priorité de conversion en bloc prompt
   360→
   361→**Format** : Liste structurée des composants avec caractéristiques.
   362→
   363→---
   364→
   365→#### Étape 3 — EXTRACTION
   366→
   367→**Objectif** : Extraction des patterns récurrents et éléments réutilisables.
   368→
   369→**Sorties attendues** :
   370→- Patterns de conception identifiés
   371→- Structures de données communes
   372→- Workflows récurrents
   373→- Contraintes techniques récurrentes
   374→- Bonnes pratiques observées
   375→- Exemples concrets dans **4+ domaines différents**
   376→
   377→**Format** : Catalogue des patterns avec exemples concrets.
   378→
   379→---
   380→
   381→#### Étape 4 — STRUCTURATION
   382→
   383→**Objectif** : Organisation des éléments extraits en structure PEK cohérente.
   384→
   385→**Sorties attendues** :
   386→- Mapping de chaque élément aux blocs A-J
   387→- Relations entre blocs
   388→- Ordre recommandé d'utilisation
   389→- Blocs obligatoires vs optionnels
   390→- Plan d'assemblage du kit
   391→
   392→**Format** : Structure hiérarchique avec mappage aux blocs.
   393→
   394→---
   395→
   396→#### Étape 5 — GÉNÉRATION
   397→
   398→**Objectif** : Rédaction du contenu complet pour chaque bloc sélectionné.
   399→
   400→**Sorties attendues** :
   401→- Contenu détaillé et opérationnel par bloc
   402→- Exemples concrets variés (4+ domaines)
   403→- Placeholders pour les variables
   404→- Cas d'erreur et traitements
   405→- Rédaction dans la langue configurée
   406→
   407→**Format** : Contenu complet de chaque bloc en Markdown structuré.
   408→
   409→---
   410→
   411→#### Étape 6 — VALIDATION
   412→
   413→**Objectif** : Évaluation qualité et complétude du kit généré.
   414→
   415→**Critères évalués** :
   416→- **Complétude** (10 pts) : Tous les blocs sélectionnés présents et complets ?
   417→- **Cohérence** (8 pts) : Références entre blocs correctes ?
   418→- **Qualité** (7 pts) : Exemples pertinents et variés (4+ domaines) ?
   419→
   420→**Sorties attendues** :
   421→- Note par critère avec commentaire justifié
   422→- Points d'amélioration prioritaires
   423→
   424→**Format** : Rapport de validation structuré avec scores.
   425→
   426→---
   427→
   428→#### Étape 7 — AMÉLIORATION
   429→
   430→**Objectif** : Corrections et optimisations finales basées sur la validation.
   431→
   432→**Actions** :
   433→- Correction des incohérences détectées
   434→- Enrichissement des exemples (objectif 4+ domaines)
   435→- Optimisation des formulations
   436→- Ajout des éléments manquants
   437→- Production du kit final versionné
   438→
   439→**Format** : Kit PEK final complet en Markdown, prêt à l'emploi, avec numéro de version.

   443→### 5.3 System Prompt du Pipeline
   444→
   445→```
   446→Tu es un Architecte en Prompt Engineering spécialisé en conversion de projets IA
   447→en kits structurés réutilisables.
   448→
   449→Méthodologie: Chain-of-Thought en 7 étapes
   450→- Étape 1: COMPRÉHENSION — Analyse approfondie du projet
   451→- Étape 2: DÉCOMPOSITION — Découpage en composants
   452→- Étape 3: EXTRACTION — Identification des patterns réutilisables
   453→- Étape 4: STRUCTURATION — Organisation en blocs PEK
   454→- Étape 5: GÉNÉRATION — Rédaction du contenu des blocs
   455→- Étape 6: VALIDATION — Évaluation qualité et complétude
   456→- Étape 7: AMÉLIORATION — Corrections et optimisations finales
   457→
   458→Contraintes:
   459→1. Préservation de l'original — Ne jamais altérer le sens du projet source
   460→2. Complétude des fichiers — Chaque bloc sélectionné doit être exhaustif
   461→3. Cohérence des références — Les cross-références entre blocs doivent être exactes
   462→4. Respect des schemas — Suivre les formats définis pour chaque bloc
   463→5. Application du Chain-of-Thought — Raisonnement étape par étape, logique et traçable
   464→6. Qualité des exemples (4+ domaines) — Varier les domaines d'application
   465→7. Automatisation et versionnage — Prévoir l'intégration en pipeline CI/CD
   466→8. Pas de greenwashing — Pas de fausses promesses, rester factuel et mesurable
   467→9. Intégration du feedback — Prévoir des mécanismes de retour et d'itération
   468→```
   469→
   470→### 5.4 Construction du Message Utilisateur
   471→
   472→Pour chaque étape N, le message utilisateur est construit ainsi :
   473→
   474→```
   475→## Contexte de la session
   476→{session.context}
   477→
   478→## Matériel source
   479→{session.sourceMaterial}
   480→
   481→## Résultats attendus
   482→{session.results}
   483→
   484→## Contraintes additionnelles
   485→{session.constraints}
   486→
   487→## Résultats des étapes précédentes
   488→
   489→### Étape 1 — COMPRÉHENSION
   490→{output_étape_1}
   491→
   492→### Étape 2 — DÉCOMPOSITION
   493→{output_étape_2}

   495→...

   499→{instruction_de_l_étape_courante}

   502→### 5.5 Appel LLM
   503→
   504→```typescript
   505→const zai = await ZAI.create();
   506→const completion = await zai.chat.completions.create({
   507→  messages: [
   508→    { role: 'assistant', content: systemPrompt },
   509→    { role: 'user', content: userMessage },
   510→  ],
   511→  thinking: { type: 'disabled' },
   512→});
   513→```
   514→
   515→> **Note critique** : Le system prompt est passé en tant que `role: 'assistant'` (convention du SDK), et le message utilisateur réel en `role: 'user'`.
   516→
   517→### 5.6 Gestion des Erreurs dans le Pipeline
   518→
   519→- Si une étape échoue : les étapes restantes sont marquées `skipped`, la session passe en `status: 'error'`.
   520→- La réponse contient les steps réussis + l'erreur.
   521→- Le client (polling) détecte le statut `error` et arrête le polling.

   551→## 7. API REST — Endpoints
   552→
   553→### 7.1 Sessions
   554→
   555→#### `GET /api/sessions`
   556→
   557→**Description** : Liste toutes les sessions avec leurs steps et validations.
   558→
   559→**Réponse** :
   560→```json
   561→{
   562→  "success": true,
   563→  "data": [
   564→    {
   565→      "id": "cuid...",
   566→      "title": "Kit Agent de Support Client",
   567→      "status": "completed",
   568→      "steps": [...],
   569→      "validations": [{ "totalScore": 22, ... }],
   570→      "createdAt": "2025-01-15T10:30:00.000Z"
   571→    }
   572→  ]
   573→}
   574→```
   575→
   576→#### `POST /api/sessions`
   577→
   578→**Description** : Crée une nouvelle session avec 7 étapes par défaut (status: pending).
   579→
   580→**Corps** :
   581→```json
   582→{
   583→  "title": "string (requis, min 3 car.)",
   584→  "context": "string (description)",
   585→  "sourceMaterial": "string (contenu source)",
   586→  "results": "string (résultats attendus)",
   587→  "constraints": "string (contraintes)",
   588→  "inputFormat": "markdown | json | yaml | code | url | pdf",
   589→  "selectedBlocks": "A,C,E,G,H",
   590→  "complexity": "simple | moyen | complexe",
   591→  "language": "fr | en"
   592→}
   593→```
   594→
   595→**Comportement** : Si aucun `projectId` n'est fourni, un projet par défaut est créé automatiquement.
   596→
   597→**Réponse** : `{ success: true, data: { id, title, steps: [...7 steps], ... }, status: 201 }`
   598→
   599→#### `GET /api/sessions/[id]`
   600→
   601→**Description** : Récupère une session avec ses steps et validations.
   602→
   603→#### `DELETE /api/sessions/[id]`
   604→
   605→**Description** : Supprime une session en cascade (steps + validations).
   606→
   607→#### `POST /api/sessions/[id]/execute`
   608→
   609→**Description** : **ENDPOINT CRITIQUE** — Exécute le pipeline CoT complet (7 appels LLM séquentiels).
   610→
   611→**Comportement** :
   612→1. Charge la session avec ses steps
   613→2. Construit le system prompt PEK
   614→3. Pour chaque step (1→7) :
   615→   a. Marque le step comme `running` en DB
   616→   b. Construit le user message avec contexte + sorties précédentes
   617→   c. Appelle le LLM via `z-ai-web-dev-sdk`
   618→   d. Sauvegarde le output et la durée en DB
   619→   e. Accumule le output pour les étapes suivantes
   620→4. Après l'étape 6 : calcule et sauvegarde la validation
   621→5. Marque la session comme `completed`
   622→
   623→**Erreurs** : Si une étape échoue, les suivantes sont marquées `skipped`, la session passe en `error`.
   624→
   625→#### `GET /api/sessions/[id]/steps/[stepNumber]`
   626→
   627→**Description** : Récupère le statut et la sortie d'une étape (utilisé pour le **polling**).
   628→
   629→**Paramètres** : `stepNumber` de 1 à 7.
   630→
   631→**Réponse** :
   632→```json
   633→{
   634→  "success": true,
   635→  "data": {
   636→    "id": "cuid...",
   637→    "stepNumber": 3,
   638→    "name": "EXTRACTION",
   639→    "status": "running",    // pending | running | completed | error | skipped
   640→    "output": "...",
   641→    "duration": 12
   642→  }
   643→}
   644→```
   645→
   646→#### `PATCH /api/sessions/[id]/steps/[stepNumber]`
   647→
   648→**Description** : Mise à jour manuelle du statut/sortie d'une étape.
   649→
   650→### 7.2 Templates
   651→
   652→#### `GET /api/templates`
   653→
   654→**Description** : Liste tous les templates (defaults en premier).
   655→
   656→#### `POST /api/templates`
   657→
   658→**Description** : Crée un template personnalisé.
   659→
   660→**Corps** :
   661→```json
   662→{
   663→  "name": "string (requis)",
   664→  "description": "string",
   665→  "systemPrompt": "string (requis)",
   666→  "blocks": "A,C,E,G,H"
   667→}
   668→```
   669→
   670→#### `GET /api/templates/[id]` / `DELETE /api/templates/[id]`
   671→
   672→**Description** : Récupère / Supprime un template.
   673→
   674→### 7.3 Validation

   677→

   679→
   680→**Corps** :

   686→```
   687→
   688→**Réponse** :
   689→```json
   690→{
   691→  "success": true,
   692→  "data": {

   694→    "coherence": 7,      // 0-8
   695→    "quality": 6,        // 0-7
   696→    "totalScore": 21,    // 0-25

   699→    "checks": [
   700→      { "label": "Complétude des blocs (4/5)", "passed": true },
   701→      ...
   702→    ],
   703→    "passedCount": 10,
   704→    "totalChecks": 12,
   705→    "passed": true
   706→  }

   841→## 9. Système de Validation & Scoring
   842→
   843→### 9.1 Barème
   844→
   845→| Critère | Points Maximum | Description |
   846→|---------|:--------------:|-------------|
   847→| **Complétude** | 10 | Tous les blocs sélectionnés présents et remplis |
   848→| **Cohérence** | 8 | Cohérence interne et logique entre blocs |
   849→| **Qualité** | 7 | Clarté, précision, utilité du contenu |
   850→| **TOTAL** | **25** | |
   851→
   852→### 9.2 Évaluation Automatique (Étape 6 du Pipeline)
   853→
   854→Après l'étape 6 (VALIDATION), le système calcule :
   855→
   856→1. **Complétude** : Compte les blocs mentionnés dans la sortie de validation (`Bloc X`, `bloc X`, `X:`, `X :`) / blocs sélectionnés × 10.
   857→2. **Cohérence** : Par défaut 8 (évaluation LLM).
   858→3. **Qualité** : Par défaut 7 (évaluation LLM).
   859→4. **Total** : Somme des trois.
   860→
   861→### 9.3 Évaluation Algorithmique (`POST /api/validate`)
   862→
   863→Validation déterministe sans LLM :
   864→
   865→- **Complétude** : Détection de patterns textuels pour chaque bloc dans le contenu.
   866→- **Cohérence** : Vérifie la présence de headings (`###`, `##`), séparateurs (`---`), couverture de blocs ≥ 80%.
   867→- **Qualité** : Vérifie la présence d'exemples, la longueur (>500, >2000), la présence de code blocks.
   868→
   869→### 9.4 Checks de Validation
   870→
   871→Liste des contrôles automatiques (12 items) :
   872→
   873→1. Complétude des blocs (X/N)
   874→2. Références croisées cohérentes
   875→3. Exemples dans 4+ domaines
   876→4. Préservation du contenu original
   877→5. Fidélité au projet source
   878→6. Structuration conforme PEK
   879→7. Variables et placeholders définis
   880→8. Cas d'erreur couverts
   881→9. Instructions claires et non ambigües
   882→10. Formats de sortie spécifiés
   883→11. Pas de greenwashing détecté
   884→12. Mécanismes de feedback prévus
   885→
   886→### 9.5 Grades
   887→
   888→| Score | Grade | Label | Couleur |
   889→|:-----:|:-----:|-------|---------|
   890→| ≥ 22 | A | Excellent, prêt à l'export | `emerald-500` |
   891→| 20-21 | B | Acceptable, améliorations recommandées | `amber-500` |
   892→| 16-19 | C | Moyen | — |
   893→| < 16 | D | Insuffisant, régénération requise | `red-500` |

   984→## 12. Flux de Données End-to-End
   985→
   986→### 12.1 Création de Session + Pipeline
   987→
   988→```
   989→[Utilisateur remplit le formulaire]
   990→         │
   991→         ▼
   992→[Validation Zod côté client]
   993→         │
   994→         ▼
   995→[POST /api/sessions] ──→ [Prisma : Project.create + Session.create + 7× Step.create]
   996→         │                        │
   997→         │                        ▼
   998→         │               [Response : session avec steps]
   999→         │
  1000→         ├──→ [pollStepStatus(sessionId)]  ← Boucle asynchrone (toutes les 2s)
  1001→         │         │
  1002→         │         ▼
  1003→         │    [GET /api/sessions/{id}/steps/{n}] × 7
  1004→         │         │
  1005→         │         ▼
  1006→         │    [Mise à jour du store : steps[], stepOutputs[], status]
  1007→         │
  1008→         ▼
  1009→[POST /api/sessions/{id}/execute]  ← Non-bloquant
  1010→         │
  1011→         ▼
  1012→[Pour chaque étape 1→7 :]
  1013→    ├── [UPDATE Step → status: 'running']
  1014→    ├── [UPDATE Session → currentStep: N]
  1015→    ├── [ZAI.create() → zai.chat.completions.create()]
  1016→    │        └── messages: [system_prompt + accumulated_context]
  1017→    ├── [UPDATE Step → status: 'completed', output, duration]
  1018→    └── [Si étape 6 : computeAndSaveValidation()]
  1019→              └── [Prisma : Validation.create]
  1020→         │
  1021→         ▼
  1022→[UPDATE Session → status: 'completed']
  1023→         │
  1024→         ▼
  1025→[Response → parse score depuis output étape 6]
  1026→         │
  1027→         ▼
  1028→[Store : setValidationScore, setValidationDetails, setIsGenerating(false)]
  1029→```

  1053→## 13. Règles d'Implémentation Inviolables
  1054→
  1055→### 13.1 Règles Architecturales
  1056→
  1057→1. **Un seul route visible** : Toute l'UI est dans `src/app/page.tsx` (route `/`). Jamais de routes additionnelles.
  1058→2. **API Routes uniquement** : Utiliser `app/api/` pour toute logique serveur. Pas de Server Actions.
  1059→3. **z-ai-web-dev-sdk côté serveur UNIQUEMENT** : Jamais d'import SDK côté client.
  1060→4. **Port 3000 unique** : Le dev server Next.js tourne exclusivement sur le port 3000.
  1061→5. **Pas de `bun run build`** : Utiliser uniquement `bun run dev` et `bun run lint`.
  1062→
  1063→### 13.2 Règles de Code
  1064→
  1065→6. **TypeScript strict** : Typage explicite pour toutes les interfaces, paramètres et retours.
  1066→7. **Composants shadcn/ui** : Toujours privilégier les composants existants (`src/components/ui/`). Ne jamais reconstruire.
  1067→8. **Imports** : Utiliser `@/` pour les chemins relatifs au projet.
  1068→9. **Pas de test code** : Ne jamais écrire de fichiers de test.
  1069→10. **Store comme source de vérité UI** : Toute la state UI passe par le Zustand store.
  1070→
  1071→### 13.3 Règles UX/UI
  1072→
  1073→11. **Pas de bleu/indigo** : Palette ambre/gold uniquement pour les éléments primaires.
  1074→12. **Footer sticky** : Toujours visible en bas, poussé naturellement par le contenu.
  1075→13. **Mobile-first responsive** : Touch targets ≥ 44px, grilles adaptatives.
  1076→14. **Semantic HTML** : Utiliser `main`, `header`, `nav`, `footer`, `section`.
  1077→15. **Dark mode par défaut** : `defaultTheme="dark"` dans le ThemeProvider.
  1078→16. **Accessibilité** : `sr-only` pour les labels d'icônes, ARIA labels sur les boutons.
  1079→
  1080→### 13.4 Règles Métier PEK
  1081→
  1082→17. **Blocs A et H obligatoires** : Ils ne peuvent jamais être désélectionnés.
  1083→18. **Minimum 2 blocs** : Une session ne peut pas être lancée avec moins de 2 blocs.
  1084→19. **Chain-of-Thought vrai** : Chaque appel LLM reçoit les sorties accumulées des étapes précédentes. Jamais d'appels parallèles dans le pipeline.
  1085→20. **Scoring /25** : La validation est toujours sur 25 points (10 + 8 + 7).
  1086→21. **Langue française** : L'interface et les contenus par défaut sont en français.
  1087→22. **Pas de greenwashing** : Les prompts système interdisent les fausses promesses.
  1088→23. **4+ domaines d'exemples** : Chaque kit doit inclure des exemples variés dans au moins 4 domaines.

  1111→## 15. Matrice de Diagnostic
  1112→
  1113→Utilisez cette matrice pour diagnostiquer et résoudre les problèmes.
  1114→
  1115→### 15.1 Problèmes Frontend
  1116→
  1117→| Symptôme | Cause Probable | Solution |
  1118→|----------|---------------|----------|
  1119→| Page blanche | Erreur d'import/hydration | Vérifier `dev.log`, corriger les imports manquants |
  1120→| Onglets ne changent pas | `setActiveTab` non appelé | Vérifier le store Zustand et les Event Handlers |
  1121→| Pipeline ne se lance pas | Validation Zod échouée | Vérifier les min lengths (title:3, description:10, sourceContent:20) |
  1122→| Steps restent "pending" | Polling non démarré | Vérifier `pollStepStatus()` dans `pek-session-form.tsx` |
  1123→| Score non affiché | Regex de parsing échouée | Vérifier le format `{N}/25` dans l'output de l'étape 6 |
  1124→| Footer flottant | Missing `min-h-screen flex flex-col` | Ajouter sur le wrapper racine dans `page.tsx` |
  1125→| Animations saccadées | Framer Motion key manquant | Vérifier `key={activeTab}` sur `AnimatePresence` |
  1126→
  1127→### 15.2 Problèmes Backend
  1128→
  1129→| Symptôme | Cause Probable | Solution |
  1130→|----------|---------------|----------|
  1131→| `POST /api/sessions` 500 | Schema Prisma non synchronisé | Exécuter `bun run db:push` |
  1132→| LLM timeout | Réponse trop longue | Le SDK gère le timeout nativement |
  1133→| Score validation = 0 | Aucun bloc détecté | Vérifier que `selectedBlocks` est bien en format "A,C,E" |
  1134→| Steps non créés | Erreur dans la création Prisma | Vérifier le `create` nested dans `sessions/route.ts` |
  1135→| Session stuck "in_progress" | Pipeline crashé silencieusement | Vérifier les logs serveur, le status doit passer à "error" |
  1136→
  1137→### 15.3 Problèmes de Données
  1138→
  1139→| Symptôme | Cause Probable | Solution |
  1140→|----------|---------------|----------|
  1141→| Anciennes sessions avec score null | Validation non encore implémentée | Le fallback dans le composant gère les null |
  1142→| Blocs obligatoires désélectionnés | `toggleBlock` bypassé | Vérifier la logique `if (config?.required) return state` |
  1143→| Store non réinitialisé | `resetPipeline()` non appelé | Toujours appeler avant une nouvelle session |
