    34→## 2. PHILOSOPHIE DE L'ÉCOSYSTÈME
    35→
    36→### Principe Fondamental
    37→**Performance-driven selection** : Le choix entre skill, agent spécialisé ou agent général est dicté par le gain de performance, pas par une hiérarchie rigide.
    38→
    39→### Modèle à Deux Couches
    40→```
    41→SKILL (protocole + connaissance domaine)
    42→    ↓
    43→AGENT SPÉCIALISÉ (exécution + outils)
    44→```
    45→
    46→- Le **skill** fournit le "comment" (protocole, templates, critères qualité)
    47→- L'**agent** fournit le "qui" (capacités spécialisées, outils, vitesse)
    48→
    49→### Principes Directeurs
    50→1. **Read before planning** — Toujours lire le projet avant de planifier
    51→2. **Performance-driven** — Sélectionner l'agent optimal pour chaque sous-tâche
    52→3. **Serial by default** — Exécution sérielle obligatoire sauf exception
    53→4. **Visible progress** — L'utilisateur sait toujours où on en est
    54→5. **Quality gates** — Validation après chaque phase
    55→
    56→### Exécution Sérielle par Défaut
    57→- **Toutes les tâches s'exécutent UNE À LA UNE** dans l'ordre défini
    58→- **Pas d'agents simultanés** sauf demande explicite + indépendance prouvée
    59→- **Quality gate entre phases** : vérifier les outputs avant de continuer
    60→- **Corrections en vol** : réévaluer si l'état du projet change significativement

   266→## 4. SKILL GEN-PLAN — Orchestration (v1.1.0)
   267→
   268→**Déclencheurs** : `gen-plan:`, `plan d'actions`, `orchestre`
   269→
   270→### 4.1 Modes
   271→
   272→| Mode | Déclencheur | Comportement |
   273→|------|-------------|--------------|
   274→| **TÂCHE** | `gen-plan: <description>` | Planifie et exécute une tâche spécifique |
   275→| **PROJET** | `gen-plan:correct-work(projet)` | Analyse complète, corrections, mise à jour prompt-master |
   276→| **CIBLE** | `gen-plan:correct-work(<cible>)` | Vérification/correction d'un élément spécifique |
   277→
   278→### 4.2 Protocole — 13 Étapes Séquentielles
   279→
   280→#### Étape 1 — Collecte et Analyse des Demandes
   281→**Objectif** : Relire toute la conversation pour extraire chaque demande explicite et implicite.
   282→- Relire tous les messages utilisateur (demandes initiales, specs, contraintes, corrections)
   283→- Identifier demandes explicites (ce que l'utilisateur a directement demandé)
   284→- Identifier demandes implicites (prérequis, effets de bord, conséquences logiques)
   285→- Lister chaque demande sans ambiguïté
   286→- **Méthode** : Lire conversation complète. Utiliser worklog.md si disponible.
   287→
   288→#### Étape 2 — Lecture et Analyse du Projet
   289→**Objectif** : Traverser tous les fichiers du projet pour comprendre structure, architecture, dépendances, état actuel.
   290→- **Principe** : Un plan sans connaissance du projet est générique et probablement inadéquat.
   291→- Structure fichiers : arborescence, configs (package.json, tsconfig, etc.)
   292→- Code source : chaque fichier significatif
   293→- Schéma base de données (si applicable)
   294→- Dépendances : imports inter-modules, dépendances externes
   295→- Configuration : fichiers config pertinents
   296→- Documentation : README, worklog, specs, fichiers référence
   297→- **Méthode** : Utiliser Read, Glob, Grep pour traverser chaque fichier significatif.
   298→
   299→#### Étape 3 — Identification de la Nature du Projet
   300→**Objectif** : Déterminer type projet, technologies, architecture, complexité.
   301→- Type projet : web app, API, script, document, analyse, skill, etc.
   302→- Technologies : framework, langage, base de données, build tools
   303→- Architecture : monolithe, micro-services, fullstack, frontend only, backend only
   304→- Complexité : nombre fichiers, profondeur arborescence, dépendances
   305→
   306→#### Étape 4 — Identification des Objectifs
   307→**Objectif** : Lister chaque objectif explicitement, sans ambiguïté ni omission.
   308→- Lister objectifs dérivés des demandes utilisateur
   309→- Lister objectifs implicites (prérequis, effets de bord, contraintes)
   310→- Vérifier que chaque objectif est mesurable (atteint ou non)
   311→- Supprimer doublons
   312→
   313→#### Étape 5 — Décomposition en Sous-tâches Adaptées
   314→**Objectif** : Décomposer chaque objectif en sous-tâches atomiques avec ordre logique d'exécution.
   315→- Pour chaque objectif, identifier sous-tâches atomiques
   316→- Ordonner logiquement (prérequis d'abord, puis exécution, puis validation)
   317→- Adapter au type projet :
   318→  - **Fullstack** : inclure vérification API frontend-backend, schéma DB, auth
   319→  - **Frontend only** : inclure vérification responsive, accessibilité, composants
   320→  - **Backend/API** : inclure vérification endpoints, validation, sécurité
   321→  - **Document/PDF** : inclure vérification contenu, layout, cohérence données
   322→  - **Script/automation** : inclure cas limites, robustesse, gestion erreurs
   323→
   324→#### Étape 6 — Détection des Dépendances
   325→**Objectif** : Identifier dépendances entre sous-tâches et contraintes de précédence.
   326→- Dépendances séquentielles (A doit finir avant B)
   327→- Dépendances parallèles (A et B peuvent tourner simultanément)
   328→- Dépendances conditionnelles (B tourne seulement si A réussit/échoue)
   329→- Vérifier absence de dépendances circulaires
   330→- **Méthode** : Construire graphe de dépendances. Identifier chemins critiques.
   331→
   332→#### Étape 7 — Priorisation
   333→**Objectif** : Classer sous-tâches par impact sur résultat final.
   334→
   335→| Priorité | Définition |
   336→|----------|------------||
   337→| **Critique** | Échec invalide le résultat entier |
   338→| **Importante** | Impact significatif sur qualité |
   339→| **Secondaire** | Améliore résultat mais non bloquant |
   340→
   341→#### Étape 8 — Estimation des Risques
   342→**Objectif** : Pour chaque sous-tâche, identifier risques potentiels et plans de fallback.
   343→- Complexité technique : connaissance spécialisée requise ?
   344→- Ambiguïté : clairement définie ou ouverte à interprétation ?
   345→- Dépendance externe : repose sur API, service, outil qui pourrait échouer ?
   346→- Fallback : que faire si sous-tâche échoue ? Contourner, reporter, simplifier ?
   347→
   348→#### Étape 9 — Structuration du Plan
   349→**Objectif** : Produire un plan d'actions formel, complet, adapté au projet.
   350→
   351→Le plan doit inclure :
   352→- **En-tête** : nature projet, objectifs, contraintes
   353→- **Liste ordonnée d'étapes** avec IDs (1, 2-a, 2-b, 3...)
   354→  - Pour chaque étape : objectif, fichiers concernés, dépendances, priorité
   355→  - Indication parallélisme où applicable
   356→  - Critères de validation par étape
   357→- **Carte dépendances** : résumé relations étapes
   358→- **Matrice risques** : risques et plans fallback
   359→
   360→#### Étape 10 — Validation du Plan
   361→**Objectif** : Checklist :
   362→- [ ] Le plan couvre-t-il toutes les demandes utilisateur ?
   363→- [ ] Y a-t-il des étapes manquantes ?
   364→- [ ] L'ordre d'exécution est-il logique (aucune dépendance violée) ?
   365→- [ ] Le plan est-il adapté au projet (pas générique) ?
   366→- [ ] Chaque étape a-t-elle un critère de validation clair ?
   367→- [ ] Les risques principaux sont-ils couverts par des plans fallback ?
   368→
   369→Si problème trouvé, retourner à l'étape pertinente et corriger.
   370→
   371→#### Étape 11 — Test et Correction Pré-Intégration
   372→**Objectif** : Tester chaque fichier généré ou modifié AVANT de l'intégrer à l'éco-système.
   373→
   374→Pour chaque fichier candidat à l'intégration :
   375→1. **Classifier le fichier** :
   376→   - **Skill** : Fichier avec YAML frontmatter (name, description) ou contenu définissant un protocole/routine réutilisable
   377→   - **Éco-système** : Fichier de configuration, prompt maître, contexte, mémoire, agent
   378→   - **Utilitaire** : Script, outil, fichier temporaire — ne pas intégrer
   379→2. **Tester le fichier** :
   380→   - **Skill (.md avec YAML)** : Vérifier YAML valide, contenu cohérent (> 200 chars), pas de placeholders
   381→   - **Python (.py)** : Compiler avec compile() ou ast.parse() pour vérifier syntaxe
   382→   - **Shell (.sh)** : Vérifier shebang, syntaxe avec bash -n, pas de commandes destructrices
   383→   - **Markdown (.md)** : Vérifier structure, absence de contenu tronqué
   384→   - **Configuration (.json, .yaml)** : Valider format, vérifier clés obligatoires
   385→3. **Corriger les anomalies détectées** immédiatement
   386→4. **Valider la cohérence entre fichiers** (références croisées valides)
   387→
   388→#### Étape 12 — Intégration Éco-Système (SPÉCIFIQUE z.ai)
   389→**Objectif** : Si les fichiers concernent des skills ou l'éco-système, les intégrer.
   390→
   391→**Prérequis** : Étape 11 (Test et Correction) doit être terminée avec succès.
   392→
   393→1. **Si le fichier est un Skill** :
   394→   - Créer le répertoire /home/z/my-project/skills/<skill-name>/ s'il n'existe pas
   395→   - Y placer le fichier SKILL.md avec le contenu validé
   396→   - Vérifier la conformité du YAML frontmatter
   397→   - Ne pas écraser un skill existant sans confirmation
   398→2. **Si le fichier est un fichier d'Éco-système** :
   399→   - Prompt maître / Contexte → /home/z/my-project/skills/gen-plan/references/
   400→   - Configuration mémoire → /home/z/my-project/skills/gen-plan/references/
   401→3. **Validation post-intégration** :
   402→   - Vérifier que chaque fichier intégré ne casse pas les skills existants
   403→   - Vérifier qu'il n'y a pas de conflit de noms
   404→   - Mettre à jour l'inventaire des skills si nécessaire

   481→
   482→**Déclencheurs** : `vérifie ton travail`, `correct_work()`, `verify_work`
   483→
   484→**3 Modes** : PROJET | CIBLE | DIRECT

   486→**5 Étapes Séquentielles** :
   487→Skill de vérification du travail réalisé. Alias : verify-work. Version v1.0.0.
   488→DÉCLENCHEURS :
   489→- "vérifie ton travail", "vérifie tes résultats", "vérifie ton code"
   490→- "correct_work", "verify_work"
   491→3 MODES D'APPEL :
   492→- PROJET : correct_work(projet) — vérification complète avec gestion du prompt-maître. Première analyse = intégralité du projet ; appels suivants = lecture du prompt-maître existant uniquement
   493→- CIBLE : correct_work(cible) — vérification ciblée sur un livrable spécifique, sans gestion du prompt-maître
   494→- DIRECT : correct_work() — vérification rapide sans analyse approfondie
   495→PROTOCOLE 5 ÉTAPES SÉQUENTIELLES :
   496→ÉTAPE 1 — Plan d'actions (appel à gen-plan) :
   497→Appelle le skill gen-plan pour générer un plan de vérification adapté au projet. Le plan sert de base pour les étapes 2 à 5. Le plan DOIT être spécifiquement adapté au type de projet vérifié (web, Python, PDF, script, etc.).
   498→ÉTAPE 2 — Erreurs et omissions :
   499→1. Relire les spécifications initiales et vérifier chaque exigence satisfaite
   500→2. Vérifier les données factuelles (noms, chemins, versions, tailles, counts)
   501→3. Vérifier la cohérence linguistique (pas de mélange de langues incohérent)
   502→4. Vérifier les fichiers de sortie (existent, lisibles, non vides)
   503→5. Vérifier les dépendances (imports, chemins, références croisées)
   504→6. Adapter la vérification au type de projet
   505→7. Corriger chaque erreur ou omission identifiée
   506→ÉTAPE 3 — Structure du code et conflits :
   507→1. Imports circulaires
   508→2. Conflits de noms (fonctions/classes/variables en double)
   509→3. Variables non initialisées ou utilisées avant définition
   510→4. Chemins en dur non portables
   511→5. Gestion des erreurs (cas d'erreur traités ou échec silencieux ?)
   512→6. Doublons de code à factoriser
   513→7. Convention de nommage cohérente
   514→8. Conflits logiques et conditions contradictoires — utiliser un TABLEAU DE KARNAUGH pour simplifier, détecter les contradictions, vérifier la couverture complète
   515→9. Corriger chaque problème identifié
   516→ÉTAPE 4 — Vérification des interactions :
   517→Adaptée au type de projet (fullstack, frontend, backend, document, script) :
   518→1. API frontend-backend : endpoints existent, params correspondent, réponses parsées, codes erreur gérés
   519→2. Props inter-composants : types corrects, callbacks bons args, pas de props manquantes
   520→3. State management : store expose données nécessaires, actions au bon moment, pas de state mort
   521→4. Flux de données bout en bout : tracer scénario complet clic→API→store→render
   522→5. Communications entre services : ports/URLs, XTransformPort, WebSockets, timeouts
   523→6. Références croisées entre livrables (documents)
   524→7. Corriger chaque problème d'interaction
   525→ÉTAPE 5 — Cohérence des raisonnements et résultats :
   526→1. Cohérence logique (pas de saut, pas de contradiction)
   527→2. Cohérence numérique (chiffres, pourcentages, tailles)
   528→3. Cohérence temporelle (dates, versions, chronologies)
   529→4. Résultat attendu vs résultat obtenu
   530→5. Cohérence entre fichiers livrables
   531→6. Corriger toute incohérence
   532→DÉPENDANCE : Ce skill dépend de gen-plan pour son Étape 1. Si gen-plan n'est pas disponible, l'Étape 1 doit être réalisée manuellement.

   702→## 8. RÈGLES D'OR (7 règles)
   703→
   704→1. **Règle 1** : Toujours lire le projet/système avant de planifier. Cette règle fondamentale garantit que toute planification ou action est basée sur une compréhension complète du contexte existant, évitant les erreurs d'assomption et les actions inutiles ou redondantes.
   705→
   706→2. **Règle 2** : Sélectionner l'agent optimal selon la performance. Utiliser la matrice de décision performance-driven pour choisir le meilleur agent pour chaque tâche, en privilégiant toujours la combinaison skill + agent spécialisé pour un résultat optimal.
   707→
   708→3. **Règle 3** : Exécuter en mode sériel par défaut (une tâche à la fois). L'exécution séquentielle garantit la qualité et la traçabilité de chaque étape, permettant de vérifier les outputs avant de passer à la tâche suivante.
   709→
   710→4. **Règle 4** : Vérifier les outputs avant de continuer. Chaque résultat produit doit être validé avant de passer à l'étape suivante, garantissant la qualité et la cohérence de l'ensemble du processus.
   711→
   712→5. **Règle 5** : Logger chaque phase dans worklog. La traçabilité est essentielle : chaque action, décision et résultat doit être consigné dans le worklog partagé pour permettre le suivi et la coordination entre agents.
   713→
   714→6. **Règle 6** : Communiquer clairement en français. Toute communication avec François doit être en français, conformément à sa préférence linguistique exprimée dans son profil.
   715→
   716→7. **Règle 7** : Utiliser les 12 agents spécialisés quand pertinent. L'écosystème complet d'agents spécialisés doit être exploité au maximum pour tirer parti de l'expertise spécifique de chacun, plutôt que de se limiter aux agents génériques.
