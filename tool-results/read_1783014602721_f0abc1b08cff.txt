    34→    34→    34→## 2. PHILOSOPHIE DE L'ÉCOSYSTÈME
    35→    35→    35→
    36→    36→    36→### Principe Fondamental
    37→    37→    37→**Performance-driven selection** : Le choix entre skill, agent spécialisé ou agent général est dicté par le gain de performance, pas par une hiérarchie rigide.
    38→    38→    38→
    39→    39→    39→### Modèle à Deux Couches
    40→    40→    40→```
    41→    41→    41→SKILL (protocole + connaissance domaine)
    42→    42→    42→    ↓
    43→    43→    43→AGENT SPÉCIALISÉ (exécution + outils)
    44→    44→    44→```
    45→    45→    45→
    46→    46→    46→- Le **skill** fournit le "comment" (protocole, templates, critères qualité)
    47→    47→    47→- L'**agent** fournit le "qui" (capacités spécialisées, outils, vitesse)
    48→    48→    48→
    49→    49→    49→### Principes Directeurs
    50→    50→    50→1. **Read before planning** — Toujours lire le projet avant de planifier
    51→    51→    51→2. **Performance-driven** — Sélectionner l'agent optimal pour chaque sous-tâche
    52→    52→    52→3. **Serial by default** — Exécution sérielle obligatoire sauf exception
    53→    53→    53→4. **Visible progress** — L'utilisateur sait toujours où on en est
    54→    54→    54→5. **Quality gates** — Validation après chaque phase
    55→    55→    55→
    56→    56→    56→### Exécution Sérielle par Défaut
    57→    57→    57→- **Toutes les tâches s'exécutent UNE À LA UNE** dans l'ordre défini
    58→    58→    58→- **Pas d'agents simultanés** sauf demande explicite + indépendance prouvée
    59→    59→    59→- **Quality gate entre phases** : vérifier les outputs avant de continuer
    60→    60→    60→- **Corrections en vol** : réévaluer si l'état du projet change significativement

   278→   278→   278→### 4.2 Protocole — 13 Étapes Séquentielles
   279→   279→   279→
   280→   280→   280→#### Étape 1 — Collecte et Analyse des Demandes
   281→   281→   281→**Objectif** : Relire toute la conversation pour extraire chaque demande explicite et implicite.
   282→   282→   282→- Relire tous les messages utilisateur (demandes initiales, specs, contraintes, corrections)
   283→   283→   283→- Identifier demandes explicites (ce que l'utilisateur a directement demandé)
   284→   284→   284→- Identifier demandes implicites (prérequis, effets de bord, conséquences logiques)
   285→   285→   285→- Lister chaque demande sans ambiguïté
   286→   286→   286→- **Méthode** : Lire conversation complète. Utiliser worklog.md si disponible.
   287→   287→   287→
   288→   288→   288→#### Étape 2 — Lecture et Analyse du Projet
   289→   289→   289→**Objectif** : Traverser tous les fichiers du projet pour comprendre structure, architecture, dépendances, état actuel.
   290→   290→   290→- **Principe** : Un plan sans connaissance du projet est générique et probablement inadéquat.
   291→   291→   291→- Structure fichiers : arborescence, configs (package.json, tsconfig, etc.)
   292→   292→   292→- Code source : chaque fichier significatif
   293→   293→   293→- Schéma base de données (si applicable)
   294→   294→   294→- Dépendances : imports inter-modules, dépendances externes
   295→   295→   295→- Configuration : fichiers config pertinents
   296→   296→   296→- Documentation : README, worklog, specs, fichiers référence
   297→   297→   297→- **Méthode** : Utiliser Read, Glob, Grep pour traverser chaque fichier significatif.
   298→   298→   298→
   299→   299→   299→#### Étape 3 — Identification de la Nature du Projet
   300→   300→   300→**Objectif** : Déterminer type projet, technologies, architecture, complexité.
   301→   301→   301→- Type projet : web app, API, script, document, analyse, skill, etc.
   302→   302→   302→- Technologies : framework, langage, base de données, build tools
   303→   303→   303→- Architecture : monolithe, micro-services, fullstack, frontend only, backend only
   304→   304→   304→- Complexité : nombre fichiers, profondeur arborescence, dépendances
   305→   305→   305→
   306→   306→   306→#### Étape 4 — Identification des Objectifs
   307→   307→   307→**Objectif** : Lister chaque objectif explicitement, sans ambiguïté ni omission.
   308→   308→   308→- Lister objectifs dérivés des demandes utilisateur
   309→   309→   309→- Lister objectifs implicites (prérequis, effets de bord, contraintes)
   310→   310→   310→- Vérifier que chaque objectif est mesurable (atteint ou non)
   311→   311→   311→- Supprimer doublons
   312→   312→   312→
   313→   313→   313→#### Étape 5 — Décomposition en Sous-tâches Adaptées
   314→   314→   314→**Objectif** : Décomposer chaque objectif en sous-tâches atomiques avec ordre logique d'exécution.
   315→   315→   315→- Pour chaque objectif, identifier sous-tâches atomiques
   316→   316→   316→- Ordonner logiquement (prérequis d'abord, puis exécution, puis validation)
   317→   317→   317→- Adapter au type projet :
